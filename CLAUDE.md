## 工作约定

- PRD 修改必须走 iterative-prd 流程：更新内容 → 升版 → 追加变更记录 → git commit
- PRD 版本格式：v0.x（草稿），v1.0（正式版，需用户明确确认）
- 数据分析结论写入 `docs/architecture/` 目录，JSON 格式
- 系统输入以"送审版"为基准（详见 PRD 4.3.4）
- 文件类型识别需支持模糊匹配，容忍命名不一致（详见 PRD 4.3.6）

## 验证工作流

对每个已整理的项目执行以下验证：

```bash
# 1. 检查目录结构完整性
ls -R data/projects/{project_id}/

# 2. 统计文件数量
find data/projects/{project_id} -type f \( -name "*.pdf" -o -name "*.xlsx" -o -name "*.docx" \) | wc -l

# 3. 验证 meta.json 完整性
cat data/projects/{project_id}/meta.json | python -m json.tool

# 4. 对照 file_checklist.md 检查必备件
# - 项目级 7 类必备件是否齐全
# - 各子项目的 4 个子目录（工程图纸/建筑图纸/工程估算书/材料清册）是否存在

# 5. 检查文件命名规范
# - 是否包含 project_id 前缀
# - 是否符合中文命名约定
```

- 数据库：PostgreSQL
- 任务调度：Celery

## 术语表

| 术语 | 含义 |
|------|------|
| 送审版 | 专家收到的待审材料（修改前），系统输入基准 |
| 审定版 | 专家评审后的修正版本（修改后），参考标准答案 |
| 三元关联 | 专家意见 ↔ 设计文件 ↔ 引用规范 的关联结构 |
| 必备件 | 所有项目必须包含的 7 类文件 |
| 可选件 | 项目间有差异的 4 类文件 |

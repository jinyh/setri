## 1. 背景与痛点

目前经研院在评审大量10kV开关站项目时，专家面临以下核心痛点：

| # | 痛点 | 现状描述 |
|---|------|----------|
| 1 | 工程量核查重复劳动 | 防震垫多配、电缆预留过长、硬件加密重复配置等低级问题需逐项人工比对 |
| 2 | 规范引用易过时 | 材差文件、定额版本等动态依据更新频繁，人工跟踪易遗漏 |
| 3 | 图文/规实不符 | 平面布置与安全距离、图纸与清单数量一致性等需跨文档交叉核验 |
| 4 | 评审意见落实难追踪 | 评审意见发出后，修改稿是否逐条落实缺乏系统化比对手段，依赖人工逐条核查 |
| 5 | 知识传承断层 | 资深专家的评审经验以口头传授为主，缺乏结构化沉淀，新人上手周期长 |

## 2. 产品定位

**不是**：全自动审批机器。
**而是**：专家的"火眼金睛"副驾驶——处理数据比对与法规检索，专家负责决策与定性。

### 三阶段演进路线

```
Phase 1: MVP验证          Phase 2: 内网部署           Phase 3: 智能体协同
(50个项目)               (200个项目)                (全面推广)
┌─────────────┐      ┌──────────────────┐      ┌──────────────────┐
│ 硬编码规则    │  →   │ 知识图谱规则引擎   │  →   │ 多智能体协同      │
│ 基础文档解析  │      │ 技经智能审查       │      │ 全流程自动化      │
│ 双屏工作台    │      │ 意见落实半自动审查  │      │ 知识问答          │
│ 专家行为记录  │      │ 金标准知识库       │      │ 综合报告生成      │
└─────────────┘      └──────────────────┘      └──────────────────┘
```

## 3. 系统架构

系统采用五层架构，自底向上：

```
┌─────────────────────────────────────────────────────┐
│                  用户交互层                           │
│   双屏工作台 · 意见生成器 · 依据溯源 · 知识问答        │
├─────────────────────────────────────────────────────┤
│                  业务应用层                           │
│        预审模块  ·  评审模块  ·  收口模块              │
├─────────────────────────────────────────────────────┤
│                  智能引擎层                           │
│   VLM文档解析引擎 · 规则/知识图谱引擎 · 智能体引擎      │
├─────────────────────────────────────────────────────┤
│                  数据治理层                           │
│     标准规范库  ·  历史项目库  ·  专家知识库            │
├─────────────────────────────────────────────────────┤
│                  知识沉淀层                           │
│   专家反馈采集 → 行为分析 → 规则提炼 → 金标准知识库     │
└─────────────────────────────────────────────────────┘
```

### 各层说明

| 层级 | 职责 | 关键组件 |
|------|------|----------|
| 数据治理层 | 为上层提供高质量、结构化的数据底座 | 标准规范库（国网/行业标准）、历史项目库（脱敏评审资料）、专家知识库（"专家意见↔设计文件↔引用规范"三元关联） |
| 智能引擎层 | 核心计算能力 | VLM多模态解析引擎（Gemini→Qwen VL）、规则匹配/知识图谱推理引擎、多智能体调度引擎 |
| 业务应用层 | 面向评审全流程的三大业务模块 | 预审（合规性+工程量）、评审（技经+多模态交叉验证）、收口（意见落实追踪） |
| 用户交互层 | 专家日常工作界面 | 双屏联动工作台、意见生成/导出、依据溯源浮窗、知识问答助手 |
| 知识沉淀层 | 持续学习闭环 | 采集专家"采纳/忽略/修改"行为 → 分析高频模式 → 提炼新规则 → 回灌知识库 |

### 技术原则：LLM 与外部工具的职责边界

系统中 LLM/VLM 承担感知、推理编排与表达职责，所有精确计算与确定性判定通过智能体调用外部工具完成。核心数据流如下：

```
感知（VLM）          推理编排（LLM Agent）           工具执行              表达（LLM）
│                       │                         │                    │
▼                       ▼                         ▼                    ▼
文档/图纸图像  →  结构化数据（带置信度） →  决定调哪个工具、   →  数值比较、阈值判断、  →  生成人类可读
                                        组装参数              造价计算、合规校验       的评审意见文案
```

### 三条红线

| # | 原则 | 说明 | 示例 |
|---|------|------|------|
| 1 | LLM 不做算术 | 所有数值比较、阈值判断、造价计算必须通过外部工具执行，LLM 仅负责提取数值并传参 | 电缆长度 3000m vs 估算值×1.15 → 调用 compare() 工具，而非让 LLM 心算 |
| 2 | LLM 不做最终合规判定 | 合规/违规的判定逻辑硬编码在工具内部，LLM Agent 负责调度工具并转述结果，不自行裁决 | 安全距离 1800mm < 2000mm → 工具返回"违规"，LLM 生成预警文案 |
| 3 | 关键数值设置置信度阈值 | VLM 提取的关键数值需附带置信度评分，低于阈值的提取结果标记为"待人工确认"，不静默传入工具 | 提取"变压器容量=800kVA"置信度 0.95 → 正常流转；置信度 0.6 → 标黄待确认 |

### 智能体工具清单（随阶段扩展）

| 工具类别 | 工具 | 功能 | 引入阶段 |
|----------|------|------|----------|
| 数值比较 | compare(a, b, op) | 两值比较（>、<、=、≈），返回布尔结果 | Phase 1 |
| 阈值校验 | check_threshold(value, rule_id) | 按规则ID查阈值并判定是否越限 | Phase 1 |
| 造价计算 | calc_cost(items, price_table) | 按价格表计算造价，校验逻辑一致性 | Phase 2 |
| 文本比对 | diff_text(original, revised) | 原文 vs 修改稿，返回变更区域 | Phase 2 |
| 语义检索 | vector_search(query, index) | 向量相似度检索历史案例/标准条款 | Phase 2 |
| 图谱查询 | graph_query(entity, relation) | 知识图谱 SPARQL/Cypher 查询 | Phase 2 |
| CAD解析 | parse_cad(file, extract_targets) | 提取设备布局、管线走向、标注尺寸 | Phase 3 |

## 4. Phase 1：MVP搭建（50个项目）

目标：搭建一个可交付、可日常使用的最小可用产品，让评审专家在真实工作中体验"硬编码规则+基础解析+双屏工作台"的核心价值。

### 4.1 用户角色

| 角色 | 职责 | 典型用户 |
|------|------|----------|
| 评审专家 | 使用系统完成预审作业，对预警做出采纳/忽略决策，导出评审意见书 | 配电网评审工程师（3-5人试点） |
| 技经专职 | 使用系统核查概算书、造价依据，辅助工程量审查 | 技经审查人员 |
| 系统管理员 | 维护规则库、标准库，管理用户权限，查看系统运行统计 | 项目组运维人员 |

### 4.2 用户操作流程

```
┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 上传  │ →  │ 自动预审  │ →  │ 双屏作业  │ →  │ 决策标注  │ →  │ 导出报告  │
└──────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

**Step 1 — 上传**
- 专家拖入项目文件包：《可行性研究报告》(PDF/Word)、《图纸》(PDF)、《概算书》(Excel/PDF)
- 系统自动识别文档类型，显示上传清单供确认

**Step 2 — 自动预审**
- 后台执行：VLM多模态解析 → 结构化提取 → 规则匹配 → 交叉验证
- 前端显示进度条，预计处理时间 ≤ 3分钟（50页文档+5张图纸）
- 完成后推送通知，生成"风险预警清单"

**Step 3 — 双屏作业**
- 左屏：原始文档浏览器（支持PDF/图纸渲染，关键区域高亮）
- 右屏：风险预警清单（按严重程度排序：强条风险 > 合规预警 > 优化建议）
- 点击右侧任一预警条目，左侧自动跳转并高亮对应段落/图纸区域

**Step 4 — 决策标注**
- 对每条预警执行操作：
  - **采纳**：预警转化为标准评审措辞，加入待导出列表
  - **忽略**：标记为误报，记录忽略原因（可选）
  - **修改**：基于预警内容，专家自行编辑措辞后加入导出列表
  - **补充**：专家手动添加系统未检出的评审意见
- 鼠标悬停预警，浮窗显示对应国网标准原文或历史评审案例（依据溯源）

**Step 5 — 导出报告**
- 一键导出标准格式的《项目评审意见书（草稿）》(Word)
- 包含：意见编号、问题描述、整改建议、依据条款
- 支持导出前预览与二次编辑

### 4.3 数据治理

#### 4.3.1 基础数据

| 数据类别 | 内容 | 来源 |
|----------|------|------|
| 标准规范 | 10kV开关站相关国网标准、典型设计、反措要求、最新材差文件 | 经研院标准库 |
| 历史项目 | 50个开关站项目全套脱敏资料（可研报告、图纸、概算书）及对应正式评审意见书 | 经研院档案 |

#### 4.3.2 核心数据结构："专家意见 ↔ 设计文件 ↔ 引用规范"三元关联

这是整个知识库的基础数据结构。对50个项目的历史评审意见逐条建立三元关联：

```
┌──────────────┐       定位到        ┌──────────────┐       依据于        ┌──────────────┐
│   专家意见     │ ──────────────── → │   设计文件     │ ← ──────────────── │   引用规范     │
│              │                     │              │                     │              │
│ · 意见原文    │                     │ · 文档名称    │                     │ · 标准编号    │
│ · 意见类型    │                     │ · 页码/章节   │                     │ · 条款编号    │
│ · 严重程度    │                     │ · 定位区域    │                     │ · 条款原文    │
│ · 整改建议    │                     │ · 原文片段    │                     │ · 标准版本    │
└──────────────┘                     └──────────────┘                     └──────────────┘
```

单条记录示例：

```json

  "opinion_id": "OP-037-05",
  "opinion": {
    "text": "建议将堵漏剂改为防水封堵材料",
    "type": "材料选型",
    "severity": "warning"
  },
  "document": {
    "file": "可行性研究报告.pdf",
    "page": 12,
    "section": "3.2 土建工程",
    "snippet": "...采用堵漏剂进行电缆沟封堵..."

```

#### 4.3.3 三元关联构建方法（分步走）

**Step 1：人工标注种子数据**

资深专家对 10 个典型项目的评审意见逐条标注三元关联，产出约 200-300 条高质量 ground truth。此数据集用于：
- 作为 Step 2 AI 辅助标注的评测基准（从中抽取 2-3 个项目作为测试集）
- 校准 prompt 模板和检索阈值
- 建立标注规范文档，统一标注口径

**Step 2：AI辅助批量标注**

将三元关联拆解为两个可并行的子任务，分别采用最适合的 AI 方案：

```
┌─────────────────────────────────┐
│         评审意见（一条）           │
└──────────┬──────────────────────┘
           │
   ┌───────────────┼───────────────┐
   ▼                               ▼
子任务A：文件定位                    子任务B：规范匹配
(长上下文 VLM)                      (RAG 检索 + LLM)
┌─────────────────┐          ┌─────────────────────┐
│ 项目全部文档页面   │          │ 标准规范向量库         │
│ (图像) + 全部意见  │          │ (Embedding Index)    │
│ 一次调用批量定位   │          │                     │
│                 │          │ 意见文本 → Embedding  │
│ Gemini 3.0 Flash │          │ → Top-5 候选条款     │
│ 百万级上下文      │          │ → LLM 判断最佳匹配   │
└────────┬────────┘          └──────────┬──────────┘
         │                               │
         ▼                               ▼
  {文件名, 页码,                    {标准编号, 条款号,
   章节, 原文片段}                    条款原文, 置信度}
         │                               │
         └───────────────┬───────────────┘
                         ▼
           合并为完整三元关联记录（附置信度）
```

- **子任务A — 意见→设计文件定位（长上下文 VLM 批量处理）**：
  - 利用 Gemini 3.0 Flash 的百万级 token 上下文窗口
  - 将单个项目的全部文档页面（~50页）转为图像，连同该项目的全部评审意见列表，一次调用送入模型
  - Prompt 要求模型对每条意见输出：对应文件名、页码、章节标题、原文片段
  - 一次 API 调用完成一个项目所有意见的文件定位，无需逐条逐页扫描
  - 预估效率：单项目 1 次调用，40 个项目约 40 次调用

- **子任务B — 意见→引用规范（RAG 检索 + LLM 验证）**：
  - 预处理：将标准规范库按条款粒度切分，Embedding 后建立向量索引
  - 检索：每条评审意见文本作为 query，检索 Top-5 候选规范条款
  - 验证：LLM 对候选条款逐一判断相关性，选出最佳匹配并输出置信度
  - 对于专家意见中未明确引用规范的情况（常见），LLM 根据意见语义推断最可能的依据条款
  - 预估效率：批量检索，50个项目全部意见可在数小时内完成

**Step 3：人工校验入库**

专家通过标注工具审核 AI 生成的候选关联：
- 高置信度（≥ 0.85）：快速确认，预计占 60-70%
- 中置信度（0.6-0.85）：重点审核，修正后入库
- 低置信度（< 0.6）：人工重新标注
- 预估效率：AI 辅助后，人工校验工作量降低约 60-70%

此数据集的价值：
- **Phase 1**：直接驱动依据溯源功能（UI-03），每条预警可追溯到具体文件位置和规范条款
- **Phase 2**：作为知识图谱构建的种子数据，三元关联自然映射为图谱中的节点与边
- **Phase 3**：作为智能体的检索语料库，支撑历史案例向量检索与知识问答

#### 4.3.4 数据基准说明（v1.1 新增）

> 本节基于对 3 个测试项目（崇明盛世新苑、奉贤悦景名筑、奉贤咉懿佳园）真实数据的差异分析得出。

系统输入以**"送审版"**目录结构为基准（即测试数据中的"修改前"版本）。

- **理由**：送审版是专家收到的待审材料，是评审流程的起点。系统的核心场景是辅助专家对送审材料进行预审，因此数据模型、文件解析、规则匹配均以送审版结构为准。
- **"审定版"（修改后）的用途**：作为参考标准答案，用于理解目录语义、验证规则有效性、训练模型。审定版反映了专家评审后的修正结果（如"送审版→审定版"重命名、"可研报告→初设报告"术语升级、"工程估算书→工程概算书"阶段变更），这些变更模式本身也是有价值的评审知识。
- **数据规模参考**：送审版总目录数 301 条，审定版总目录数 280 条。三个项目的子目录数分别为 79/93/87（送审版）和 81/93/64（审定版）。

系统在文档上传阶段应区分必备件与可选件：必备件缺失时给出强提醒，可选件缺失仅做弱提示。

#### 4.3.6 命名规范化需求（v1.1 新增）

> 基于 3 个测试项目的目录命名对比分析，发现 13 类命名不一致问题。系统需要建立同义词映射表，支持模糊匹配。

**问题 1：分隔符不统一**

项目1/2 使用空格分隔子工程名，项目3 使用括号：
- 项目1：`...盛世新苑10kV普通住宅红线内接入工程 内线电缆-0.4kV`
- 项目3：`...咉懿佳园10kV普通住宅红线内接入工程（0.4kV电缆）`

**问题 2：同一概念多种叫法（7组同义词映射）**

| 标准概念 | 变体1 | 变体2 | 变体3 |
|----------|-------|-------|-------|
| 低压电缆 | 内线电缆-0.4kV | 0.4kV电缆 | （0.4kV电缆） |
| 高压电缆 | 内线电缆-10kV | 10kV电缆 | （10kV电缆） |
| 信息采集 | 信息采集 | 用电信息采集 | （信息采集） |
| 通信 | 内线通信 | 通信 | （通信） |
| 其他依据 | 其它依据 | 其他 | 其他依据 |
| 内审意见 | 内审意见 | 内审意见及签到单 | — |
| 设计委托书 | 设计中标通知书或委托书 | 设计委托书 | — |

**问题 3：括号全角/半角混用**

- 半角：`工程估算书(PDF版)`（项目1、项目2送审版）
- 全角：`工程概算书（PDF版）`（项目3）

**问题 4：阶段术语混用**

送审版与审定版之间存在术语升级，不同项目处于不同阶段：
- 报告名称：`可研报告` vs `初设报告`（项目2 审定版从可研升级为初设）
- 概算名称：`工程估算书` vs `工程概算书`（项目2 审定版从估算升级为概算）
- 图纸名称：`其他图纸` vs `图纸资料` vs `设计图纸`

**系统应对策略**：
1. 建立**同义词映射表**，将上述变体统一映射到标准概念
2. 文件类型识别采用**模糊匹配**策略，容忍分隔符、括号、措辞差异
3. 括号匹配时同时支持全角 `（）` 和半角 `()`
4. 阶段术语（可研/初设、估算/概算）视为同一文件类型的不同阶段表述

### 4.4 文档解析引擎（VLM技术路线）

本系统不走传统 OCR + 后处理管线，而是直接采用视觉语言模型（VLM）进行端到端的文档理解，分两步在 Phase 1 内完成：

```
Phase 1a: Gemini 3.0 Flash (API)        Phase 1b: Qwen VL (微调私有化)
┌───────────────────────────┐        ┌───────────────────────────┐
│ · 调用 Gemini API 快速验证  │   →    │ · 基于50个项目数据微调      │
│ · 跑通全链路原型            │        │ · 内网GPU服务器私有化部署    │
│ · 积累 prompt 模板与评测集  │        │ · 脱离外部API依赖           │
│ · 标注模型输出质量基线      │        │ · 达到Gemini同等或更优效果   │
└───────────────────────────┘        └───────────────────────────┘
```

**为什么选 VLM 而非传统 OCR**：
- 传统 OCR 管线（OCR → 版面分析 → 表格识别 → 后处理）环节多、误差累积、维护成本高
- VLM 端到端理解文档图像，天然支持文本、表格、图纸的统一处理
- 对复杂排版（合并单元格、跨页表格、图文混排）的鲁棒性更强
- Prompt 驱动的抽取方式，新增字段无需重新训练管线

**Gemini 3.0 Flash 阶段（Phase 1a）**：
- 将文档页面转为图像，直接输入 Gemini 3.0 Flash
- 通过结构化 Prompt 指令提取：章节结构、关键参数、BOM清单、图纸标注
- 输出 JSON 格式的结构化数据，供规则引擎消费
- 同步建立评测数据集（50个项目的人工标注 ground truth）

**Qwen VL 微调阶段（Phase 1b）**：
- 基于 Phase 1a 积累的 prompt 模板 + 人工校正数据，构建微调训练集
- 在 Qwen VL 基座上进行领域微调（LoRA / 全量微调）
- 部署于内网 GPU 服务器，消除对外部 API 的依赖
- 验收标准：在评测集上达到 Gemini 3.0 Flash 同等或更优的抽取效果

**功能清单**

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| F-01 | 文档结构化解析 | VLM端到端识别 + 章节自动分割（系统、一次、土建、技经），提取关键参数（变压器容量/数量、电缆型号等） | P0 |
| F-02 | BOM清单提取 | 精准识别概算表/物料清单中的设备名称、规格型号、数量、单价 | P0 |
| F-03 | 图纸关键信息提取 | 识别一次系统图拓扑、平面布置图设备布局；重点：开关柜数量、变压器数量、标注尺寸 | P1 |
| F-04 | 签字盖章检测 | VLM 识别设计文件封面页/审批页中的签名有无、印章有无及印章文字内容，输出结构化结果供 R-10/R-11 规则消费 | P1 |

### 4.5 硬编码规则库

**技经与工程量核查**

| ID | 规则 | 逻辑/阈值 | 提示文案 |

| R-01 | 防震垫配置核查 | 变压器数量 × 4 > 实际需要（通常按2组/台） | 检测到变压器防震垫配置为4套，建议核减至2套（依据典型设计） |
| R-02 | 加密装置冗余查重 | 单站"硬件加密装置"数量 > 1 | 开关站通常仅需一套硬件加密装置，当前清单为2套，建议核减 |
| R-03 | 控制电缆长度预警 | 清单长度 > 图纸路径估算 × 1.15 | 控制电缆清单长度远超图纸估算，建议核实是否冗余 |

## 5. Phase 2：内网部署（200个项目）

### 5.1 数据治理升级

- 历史项目库扩展至 200个 脱敏开关站项目

- 设计 动态知识更新机制：支持增量式图谱更新、版本控制与知识演化追踪

### 5.2 知识图谱替代硬编码

基于"评审规则知识图谱构建 → 专家意见语义匹配 → 多层次推理算法"三步走：

| 能力 | 指标 |
|------|------|
| 知识图谱节点规模 | ≥ 500个（覆盖核心技术标准与管理规定） |
| 规则匹配准确率 | ≥ 85% |
| 单次匹配响应时间 | < 2秒 |
| 多层次规则处理 | 支持总则-分则-细则层级，一条意见对应多个规则点 |
| 规则冲突预警 | 匹配到矛盾规则时自动预警 |

核心技术：
- NER + 关系抽取 → 结构化三元组 → 图数据库存储
- Sentence-BERT / 图神经网络(GNN) 实现语义匹配
- 层次化匹配算法处理复合意见的多步推理

### 5.3 技经智能审查

| 功能 | 描述 |
|------|------|
| 造价文件自动解析 | 从设计说明书、预算清单、施工方案中提取关键字段（电压等级、容量配置、材料清单） |

- 文档布局识别：基于 Qwen VL 微调模型，理解复杂排版文档的结构
- 表格智能抽取：支持合并单元格、跨页表格、嵌套表格的精准提取
- 图示理解：从工程图纸中提取设备拓扑关系、标注参数、布局信息

### 5.5 意见落实半自动审查

- 文本比对：原评审意见 vs 修改稿，自动标注变更区域
- 语义匹配：将每条评审意见与修改内容进行语义关联，判断是否已落实

目标：构建多智能体协同架构，实现预审—评审—收口全流程的深度自动化，配合知识问答系统，形成完整的智能评审平台。

- 任务调度：根据上传文档类型，自动分配子智能体执行顺序
- 冲突检测：当多个智能体输出矛盾结论时，标记冲突并提请专家裁决
- 综合报告生成：汇总各智能体结果，生成完整的《智能预审报告》

## 1. 文档定位与依据

本文件是统一交付演示系统的主 PRD，用于同时服务两类对象：

- 甲方：理解系统定位、交付边界、阶段范围和验收口径
- 后续 AI/研发：按统一模块边界、对象约束和页面结构继续实现

本文件不再承担旧版 PRD 中的大量底层研究过程描述，也不作为所有 schema、目录树、映射表的唯一载体。详细结构化约束以下沉文档承载，本文件只保留系统级定义和实现所需的最小约束。

本文件采用统一交付视角，不再把“开关站专项副驾驶”作为整份产品文档的主语。开关站相关能力降级为统一系统中的首阶段重点样板场景和专题能力来源。

与 P3-P5 相关的详细方法层约束以下沉设计文档承载，包括“多模态审核技术路线”“专家知识库与规则演化”“评测与执行底座工程”。这些文档用于定义知识资产、规则生成、执行 trace、评测回归和发布门槛，不与主 PRD 争抢系统级叙事职责。

## 2. 产品定位与目标

### 2.1 产品定位

统一交付演示系统定位为：

**面向经研院专家与业务人员的配电网项目智能审查/核查工作台。**

它不是 8 套独立系统的拼接集合，也不是纯算法验证台，而是一套可基于真实资料包完成“导入、解析、核查、追证、输出”的统一业务型演示系统。

### 2.2 核心目标

系统建设目标是把 `合同/` 目录下 8 份项目材料中可沉淀为产品能力的部分统一承接，形成：

- 可讲述：甲方能清晰理解统一平台与各专题成果的关系
- 可操作：用户能在浏览器内完成完整演示流程
- 可扩展：后续合同项和专题能力可持续挂接到统一架构
- 可验收：系统能够说明每项研究成果在平台中的承载位置

### 2.3 非目标

以下内容不作为本阶段 PRD 的交付承诺：

- 不按“每份合同一套独立系统”的方式建设
- 不以“全自动替代专家”为目标
- 不把论文、专利、报告等科研成果直接等价为系统功能页面
- 不在首期一次性完成全部 P3-P5 真能力
- 不覆盖全部电压等级和全部工程类型

## 3. 用户与典型场景

### 3.1 目标用户

主用户包括：

- 业扩项目评审专家
- 技经评审人员
- 开关站专项评审人员
- 竣工验收与资料核查人员
- 项目过程管控与管理分析人员
- 演示讲解、验收汇报与方案答疑人员

### 3.2 典型场景

系统至少支持以下演示与业务场景：

1. 首期从平台预注册样例项目/案例中选择资料包并触发处理，完成资料解析与基础核查；真实资料包新增导入作为后续扩展能力
2. 对项目资料执行规则匹配、逻辑校验和问题发现
3. 对技经文件执行关键字段抽取、一致性校验和问题定位
4. 对专家意见、修改内容和规则命中结果进行追踪与展示
5. 通过知识/规则中心完成问答、检索和证据追溯
6. 对进度或竣工资料专题进行独立演示
7. 在验收汇报中说明每项合同成果在系统中的承接位置

## 4. 系统总体结构

### 4.1 总体形态

统一交付演示系统采用“公共能力底座 + 核心主线模块 + 专题扩展模块”的结构。

#### 公共能力底座

- 项目与案例管理
- 多模态资料接入与解析
- 知识与规则中心
- 核查与推理引擎
- AI 助手与问答
- 结果输出与证据链展示

#### 核心主线模块

- `M1` 业扩项目智能评审
- `M2` 技经资料预审与评审

#### 专题扩展模块

- `M3` 开关站专家知识库
- `M4` 工程智慧进度管控
- `M5` 竣工资料智能核查

### 4.2 模块定义

| 模块 | 定位 | 主要承接合同主题 | 首期策略 |
|------|------|------------------|----------|
| M1 业扩项目智能评审 | 统一主线入口，承接预审、评审、收口与结果展示 | 2024 智能化审查、2025 规则自动匹配、2026 智能体协同 | 首期主要承接统一评审入口、规则来源展示与结果表达，优先消费 P1/P2 真实结果，并为智能体协同预留扩展位 |
| M2 技经资料预审与评审 | 面向技经资料的解析、校验与问题定位 | 2025 技经大模型预审、2025 技经智能辅助评审 | 首期在 P2 基础闭环之上，真实承接稳定资料集的字段抽取、基础一致性校验与问题定位 |
| M3 开关站专家知识库 | 专家经验、规则条件、知识查询与演化展示 | 2025 开关站专家知识库 | 首期保留专题骨架与承接说明，后续扩展为独立可演示专题 |
| M4 工程智慧进度管控 | 进度感知、风险预警、纠偏仿真 | 2026 智慧进度管控 | 首期保留专题骨架与数据/结果接口扩展位 |
| M5 竣工资料智能核查 | 图纸-实物-档案关联与三类核查 | 2026 竣工资料核查 | 首期保留专题骨架与规则/证据链扩展位 |

### 4.3 统一业务主线

所有模块尽量复用同一条业务主线：

1. 选择项目或案例
2. 导入资料或过程数据
3. 进行解析、抽取和结构化
4. 调用知识库、规则库或图谱进行匹配
5. 执行核查、推理或预警分析
6. 展示问题、证据与建议
7. 生成人工可确认的最终结论或报告

模块差异主要体现在第 3-6 步的处理引擎和结果解释方式，而不是入口形态。

首期在第 2 步以平台预注册样例项目/案例选择为主，不把浏览器内新增真实资料包上传作为硬验收前提。

### 4.4 能力链路与 P1-P5 的关系

`P1-P5` 不再作为 PRD 的全文主骨架，而是作为统一系统底层能力链路存在：

| 能力链路 | 作用 | 首期要求 |
|----------|------|----------|
| P1 规范库构建 | 规范源登记、扫描、摘要、产物管理 | 真实闭环 |
| P2 文件整理与解析 | 项目资料归档、路径映射、基础解析 | 真实闭环 |
| P3 专家知识库 | 专家经验、知识抽取、更新演化 | 产品骨架 |
| P4 交叉提炼 | 规则形成、规则维护、规则评估 | 产品骨架 |
| P5 自动审查 | 规则执行、结果生成、意见管理 | 产品骨架 |

其中，P5 的目标主链路是“规则库直接评审设计文件”；P3 专家知识库默认承担校准、解释增强和反馈闭环角色，不替代规则库对设计文件的主评审链路。P3-P5 的详细实现约束以 `docs/architecture/` 下的专题设计文档为准。

## 5. 首期交付范围

### 5.1 首期必须完成

首期交付必须完成以下内容：

- 统一首页与模块导航
- 项目/案例列表与任务列表
- P1 规范库基础真实闭环
- P2 文件整理基础真实闭环
- M2 基于稳定资料集的字段抽取、基础一致性校验与问题定位
- 任务状态、结果摘要、产物入口、证据/日志入口
- M1/M2 的统一结果表达方式
- P3-P5 对应专题骨架页

### 5.2 首期不做伪实现

以下能力在首期可以保留骨架，但不能用假数据伪装为已完成能力：

- 开关站专家知识库的完整知识图谱构建与动态更新
- 智能体协同总控、事实总线和多智能体自动联动
- 进度预测、风险预警和纠偏仿真真能力
- 竣工资料的跨文档实体对齐与三类自动核查真能力
- 大规模 RAG、复杂推理和全量 VLM 深解析

### 5.3 首期页面范围

统一工作台首期至少包含：

1. 系统首页
2. 模块导航页
3. 项目/案例列表页
4. 任务列表页
5. P1 工作区
6. P2 工作区
7. 任务详情页
8. 规范库浏览页
9. 结果详情页
10. M3-M5 专题骨架页

## 6. 关键业务对象与接口约束

### 6.1 关键业务对象

#### 项目/案例对象

至少包含：

- `project_id`
- `name`
- `project_type`
- `scenario_tags`
- `source_type`
- `status`
- `created_at`
- `updated_at`

首期项目与案例采用统一对象模型，`source_type` 用于区分来源而不是区分两套实体。首期推荐最小枚举：

- `sample_library`
- `admin_import`

#### 统一任务对象

所有长任务统一具备以下字段：

- `task_id`
- `pipeline_stage`
- `related_project_id`
- `status`
- `progress`
- `message`
- `started_at`
- `finished_at`
- `artifacts`
- `result_summary`

首期推荐最小枚举如下：

- `pipeline_stage`: `p1`、`p2`、`p3`、`p4`、`p5`
- `status`: `pending`、`running`、`succeeded`、`failed`

#### P1 规范库结果对象

至少包含：

- `result_id`
- `task_id`
- `source_files`
- `summary`
- `artifact_refs`
- `generated_at`

#### P2 文件整理结果对象

至少包含：

- `result_id`
- `task_id`
- `project_id`
- `path_mapping`
- `warnings`
- `summary`
- `artifact_refs`

`P2` 基础结果对象首期聚焦整理与基础解析，不把深度字段抽取和一致性校验结果作为该对象的硬要求；相关结果由 `M2` 的统一结果表达承载。

#### 结果与证据对象

统一结果页至少能表达：

- 问题或结论项
- 来源规则/知识/资料依据
- 证据片段或证据入口
- 置信度或人工确认状态
- 人工处理动作

首期证据表达采用分级口径：

- 自动结论至少能回溯到 `file + page + snippet`
- `P1/P2` 整理类结果至少提供来源文件摘要、产物入口或日志入口
- 首期“人工处理动作”以状态占位和建议动作为主，不要求提供确认、忽略、修改等写回接口

### 6.2 最小 API 分组

首期推荐 API 分组如下：

- `/api/projects`
- `/api/regulations`
- `/api/pipelines/p1`
- `/api/pipelines/p2`
- `/api/tasks`

### 6.3 首期最小接口能力

首期必须支持：

- 项目/案例列表与详情查询（统一对象模型，按 `source_type` 区分来源）
- P1 任务创建、任务查询、产物查询
- P2 任务创建、任务查询、路径映射结果查询
- 统一任务中心过滤与详情查询
- 规范库结果列表与详情查询

接口设计原则如下：

- 长任务采用“按管道创建 + 统一任务中心查询”
- 前端不直接读取底层文件系统目录
- 结果对象必须兼容后续 M3-M5 扩展

## 7. 页面与交互要求

### 7.1 首页职责

首页用于统一叙事，不承担复杂业务操作。至少展示：

- 系统定位
- 核心主线模块
- 专题扩展模块
- 合同成果统一承接说明
- 案例入口
- 当前任务概览

### 7.2 P1/P2 工作区要求

P1/P2 工作区至少包含：

- 输入区（首期以样例项目/案例选择和任务参数设置为主）
- 启动任务入口
- 任务进度区
- 结果摘要区
- 产物入口
- 证据/日志查看入口

### 7.3 统一交互原则

- AI 先行，人工兜底
- 状态可见，进度可追踪
- 结果必须可解释、可回溯
- 首期人工处理以状态展示和建议提示为主，不把写回协同动作作为硬要求
- 骨架页明确标注“当前阶段未完成能力”，不得误导为真能力
- 同一类结果在不同模块间尽量复用相同表达方式

## 8. 合同承接原则与未验收内容处理

### 8.1 合同承接原则

系统需求以 `合同/` 目录下 8 份项目材料为依据，但承接方式采用“统一平台归并”，不采用“逐合同拆成 8 套产品”。

详细对应关系见 `docs/合同承接矩阵.md`。该矩阵是合同内容到系统模块的唯一追踪文档，用于：

- 逐条确认 8 份合同的承接位置
- 标记哪些能力为首期覆盖、哪些为后续演进
- 明确未验收内容在系统中的保留位置

### 8.2 已验收与未验收内容的处理原则

- 已验收合同：在统一系统中体现其成果承接位置和复用价值，但不要求在本次重构中全部重做为真能力
- 执行中合同：必须在新版 PRD 和合同矩阵中明确模块落位、交付阶段和后续实现抓手，不能遗漏
- 招标中合同：必须在统一系统结构中预留专题入口、对象边界和结果表达方式，确保后续可无结构性返工地接入

### 8.3 不直接转译为页面的成果

以下合同成果不直接转译为独立页面或功能项，但需要在矩阵中保留说明：

- 论文
- 发明专利
- 研究报告文本本身
- 第三方测试报告本身

这些成果对应的是系统能力来源、方法验证或外部交付物，不等价于工作台中的单独业务入口。

## 9. 非功能要求与验收口径

### 9.1 通用要求

系统必须满足以下要求：

- 可解释：所有自动结论至少能回溯到资料片段、规则条款、知识节点或历史案例之一
- 可追溯：记录输入来源、字段来源、规则命中来源和结果来源
- 可扩展：允许增加新模块、新规则类型和新文档模板
- 人机协同：输出支持人工复核、确认、修正和补充
- 演示友好：支持用真实案例在有限时间内完成重点能力讲解

### 9.2 首期验收判断问题

从统一演示系统角度，首期验收至少应能回答：

1. 这是一套统一平台，而不是多份合同的散装拼接吗
2. 系统是否有清晰的统一业务主线
3. P1/P2 是否具备真实可运行闭环
4. 任务、结果、产物和证据是否可追踪
5. P3-P5 是否有清晰扩展位且不误导用户
6. 是否能够说明每项合同成果在系统中的承接位置

### 9.3 首期验收标准

首期至少满足以下标准：

- 前端、API、worker 可独立启动
- P1 可创建任务、执行任务、查询状态、查看产物
- P2 可创建任务、执行任务、查询状态、查看路径映射
- 至少 1 个预注册样例项目能跑通新 P2 链路
- M2 首期至少基于 `可研/初设报告`、`概算/估算书`、`图纸/材料清册` 三类稳定资料完成结果展示，并支持至少 1 组跨文档一致性校验
- 前端能展示任务状态、结果摘要、产物入口以及与当前结果类型匹配的证据/日志入口
- M3-M5 骨架页存在且明确标示阶段状态
- 新工程主线不再依赖旧 CLI 入口和旧占位前后端骨架

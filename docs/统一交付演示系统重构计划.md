## 1. 背景与结论

当前仓库存在明显的代际切换：

- 早期实现以 Python CLI/P1 原型为主
- 当前产品目标已切换为统一交付演示系统
- PRD v0.8 已明确从 CLI Skill 交互迁移到 Web 工作台
- 现有 `server/`、`streamlit_app/`、`tests/`、`web/` 多数为占位或中间产物
- `src/` 虽有少量代码，但整体仍属于旧阶段原型，无法作为后续主工程直接演化

因此，本次重构采用以下总原则：

1. **全量清场旧中间产物**
2. **仅迁移少量知识资产**
3. **按前后端 monorepo 重新建立主工程**
4. **第一阶段同时推进统一演示系统骨架与 P1/P2 真实能力**
5. **P3-P5 先保留产品骨架，不做伪实现**

## 2. 当前仓库评估

| 目录/模块 | 当前状态 | 评估结论 |
|------|------|------|
| `server/` | 空目录 | 直接删除 |
| `streamlit_app/` | 仅 `__init__.py` | 直接删除 |
| `tests/` | 仅空包结构 | 删除并重建 |
| `web/` | 仅占位 `package.json` 与安装产物 | 删除并重建 |
| `src/` | 早期 Python CLI + P1 包装层 | 不作为主工程保留 |
| `.claude/skills/reg-extractor/scripts/` | 存在可复用纯脚本 | 保留并择机封装 |
| `data/projects/` | 存在样例项目产物 | 作为测试夹具与演示数据使用 |
| `data/regulations/` | 存在规范库样例输出 | 作为测试夹具与演示数据使用 |

### 2.1 对旧 `src/` 的判断

`src/` 不是空壳，但不适合作为新主工程基线，原因包括：

- 入口仍是 CLI 而非服务化/API 化
- P1 之外能力缺失，P2 尚未形成真实实现
- 多处模块仍为占位
- 结构围绕旧阶段“脚本封装”组织，而非围绕新系统的领域边界组织
- 数据结构和当前 PRD 中的中文目录规范未完全对齐

结论：**不保留 `src/` 作为工程骨架，只提取其中少量可复用知识资产。**

## 3. 重构目标

本次重构的目标不是修补旧原型，而是建立一套新的统一工程基线，满足以下要求：

### 3.1 产品目标

构建一个可展示、可讲解、可扩展的统一交付演示系统，承接：

- 统一首页与模块导航
- 真实案例管理与结果展示
- P1 规范库构建能力
- P2 文件整理能力
- P3-P5 产品骨架与后续扩展位

### 3.2 工程目标

构建一套可持续演进的 monorepo 工程结构，满足：

- 前后端职责清晰
- 管道逻辑可复用
- 长任务可追踪
- 测试体系真实可运行
- 数据与业务模型有统一边界
- 后续可平滑扩展到 P3-P5 与更多专题能力

### 3.3 产品结构与统一业务主线

系统在产品层面采用“公共能力底座 + 核心主线模块 + 专题扩展模块”的结构：

- 公共能力底座
  - 项目与案例管理
  - 多模态资料接入与解析
  - 知识与规则中心
  - 核查与推理引擎
  - AI 助手与问答
  - 结果输出
- 核心主线模块
  - M1：业扩项目智能评审
  - M2：技经资料预审与评审
- 专题扩展模块
  - M3：开关站专家知识库
  - M4：工程智慧进度管控
  - M5：竣工资料智能核查

所有模块尽量复用同一条业务主线：

1. 选择项目或案例
2. 导入资料或过程数据
3. 进行解析、抽取和结构化
4. 调用知识库、规则库或图谱进行匹配
5. 执行核查、推理或预警分析
6. 展示问题、证据与建议
7. 生成人工可确认的最终结论或报告

第一阶段的重点不是一次做完这条主线的所有环节，而是先把这条主线固化为统一工作台的信息架构和 P1/P2 的真实落地点。

## 4. 总体策略

### 4.1 重构方式

采用 **“前后端 monorepo + 全量清场 + 知识资产迁移”** 的方式推进。

不采用以下路径：

- 不在旧 `src/` 上继续堆功能
- 不保留 `web/`、`server/` 的旧占位骨架
- 不为了兼容旧 CLI 而限制新架构
- 不在第一阶段强行做完 P3-P5 真能力

### 4.2 第一阶段范围

第一阶段采用“两者并进”：

- 一方面搭建统一交付演示系统骨架
- 一方面让 P1/P2 具备真实可运行的后端闭环能力

具体边界如下：

| 能力 | 第一阶段要求 |
|------|------|
| 统一首页与模块导航 | 必须完成 |
| P1 规范库 | 必须完成基础真实闭环 |
| P2 文件整理 | 必须完成基础真实闭环 |
| P3-P5 | 仅做产品骨架页与扩展位 |
| AI 助手/RAG/复杂推理 | 不作为首阶段阻塞项 |
| 全量 VLM 深解析 | 不作为首阶段阻塞项 |

### 4.3 首阶段技术默认值

下列技术项作为第一阶段默认值使用，除非后续实现中出现明确阻塞：

- 前端：Next.js（App Router）+ TypeScript
- UI：shadcn/ui + Tailwind CSS
- 后端 API：FastAPI
- 后台任务：Celery + Redis
- 持久化：文件系统 + JSON 元数据优先，数据库位预留
- 部署：Docker 化作为后续交付增强项，不作为第一阶段启动前置条件

这些默认值的目的，是减少首阶段决策摩擦；它们不是不可变更的长期架构承诺。

## 5. 新工程结构

建议建立如下 monorepo 结构：

```text
/
├── apps/
│   ├── web/                 # Next.js 工作台
│   ├── api/                 # FastAPI HTTP API
│   └── worker/              # Celery / 后台任务执行器
├── packages/
│   ├── domain/              # 领域模型、schema、枚举、共享类型
│   ├── pipelines/           # P1/P2 管道逻辑
│   └── connectors/          # 文件系统、PDF、LLM/VLM、任务队列适配层
├── tests/
│   ├── unit/
│   ├── integration/
│   ├── fixtures/
│   └── e2e/
├── data/                    # 演示样例数据、测试夹具、产物目录
├── docs/
└── scripts/
```

### 5.1 各层职责

#### `apps/web`

负责：

- 统一工作台 UI
- 首页、模块页、任务页、结果页
- 调用 API 展示任务状态和结果
- 不直接读取 `data/` 文件系统内容

#### `apps/api`

负责：

- 项目、规范库、任务、结果查询 API
- 统一任务创建入口
- 输入校验与对外协议
- 不直接承载耗时型管道执行

#### `apps/worker`

负责：

- 执行 P1/P2 长任务
- 管理任务状态、进度、产物写入
- 与 API 通过任务模型协作

#### `packages/domain`

负责：

- 统一 schema、枚举、结果对象
- 规范条款、项目元数据、任务状态、产物信息等核心模型
- 作为 API、worker、pipeline 之间的单一领域协议层

#### `packages/pipelines`

负责：

- P1/P2 业务编排
- 输入输出转换
- 可测试的纯处理逻辑
- 不混入 UI 或 HTTP 细节

#### `packages/connectors`

负责：

- 文件系统读写
- PDF/Excel/Word 接入
- 外部模型与服务适配
- 队列、缓存、持久化适配

## 6. 旧资产处理方案

### 6.1 直接删除

以下目录作为旧中间产物，直接退出主线：

- `server/`
- `streamlit_app/`
- `tests/` 现有空结构
- `web/` 现有占位工程
- `src/` 旧 CLI/包装层主结构

### 6.2 仅迁移知识资产

以下内容应保留其业务知识，但不保留原目录结构：

1. `src/setri/common/synonyms.py`
   - 迁移目录映射、专业映射、项目注册表
   - 作为 P2 分类/归档服务的基础规则源

2. `src/setri/common/schemas.py`
   - 提炼可复用的数据模型思路
   - 在新 `packages/domain` 中按新边界重写

3. `.claude/skills/reg-extractor/scripts/`
   - 迁移或封装其中已验证的纯脚本能力
   - 不再通过旧 `src/setri/p1_regulations/*` 包装方式继续使用

4. `data/projects/`、`data/regulations/`
   - 作为样例产物、测试夹具、前期演示数据
   - 不作为新目录规范的单一真相源

### 6.3 明确废弃

以下做法视为废弃，不再沿用：

- 旧 CLI 主入口
- 旧英文目录组织
- 旧 `src` 下按阶段拼接的目录方式
- 只有占位包而没有测试内容的“伪测试结构”
- 只有 npm 初始化内容的“伪前端工程”

## 7. P1/P2 第一阶段能力设计

### 7.1 P1 规范库构建

第一阶段必须完成的内容：

- 规范源登记
- 关键词扫描
- 中间结果保存
- 任务状态查询
- 结果摘要展示
- 产物访问入口

第一阶段暂不强求：

- 全量 LLM 条款抽取
- 完整冲突语义裁决
- 复杂规范版本治理

### P1 结果要求

每个 P1 任务至少输出：

- `task_id`
- 输入参数
- 状态与进度
- 扫描摘要
- 产物列表
- 可供前端展示的结果摘要对象

### 7.2 P2 文件整理

第一阶段必须完成的内容：

- 项目资料导入
- 同义词映射识别
- 标准化目录生成
- 源路径到目标路径映射记录
- 元数据写入
- 结果摘要展示

第一阶段暂不强求：

- 全量多模态解析
- 深度字段抽取
- 复杂 resolution 跟踪
- P3 直接消费的完整结构化解析结果

### P2 结果要求

每个 P2 任务至少输出：

- `task_id`
- `project_id`
- 导入来源
- 标准化目录写入结果
- `path_mapping`
- 异常与警告清单
- 可供前端展示的结果摘要对象

- 核心主线模块
- 专题扩展模块
- 案例入口
- 当前任务概览

### 8.2 P1/P2 页面职责

P1/P2 工作区至少包含：

- 输入区
- 启动任务入口
- 任务进度区
- 结果摘要区
- 产物入口
- 证据/日志查看入口

### 8.3 P3-P5 策略

P3-P5 第一阶段仅保留：

- 页面占位
- 能力范围说明
- 后续规划入口

禁止在第一阶段使用假数据伪装为已完成真能力。

## 9. 接口与数据约束

### 9.1 统一任务模型

所有长任务统一具备以下字段：

- `task_id`
- `pipeline_stage`
- `status`
- `progress`
- `message`
- `started_at`
- `finished_at`
- `artifacts`

### 9.2 API 分组

第一阶段建议 API 分组如下：

- `/api/projects`
- `/api/regulations`
- `/api/pipelines/p1`
- `/api/pipelines/p2`
- `/api/tasks`

### 9.3 存储策略

第一阶段默认采用：

- 文件系统 + JSON 元数据为主
- 数据库接口位可预留，但不作为启动阻塞项
- 若接入数据库，优先存储任务状态与检索索引，不强制所有业务内容入库

### 9.4 数据规范

统一原则：

- Web 不直接访问文件系统
- API 不直接承载长耗时业务执行
- Worker 不依赖前端实现细节
- 中文标准目录是新 P2 的唯一目录规范
- 历史英文结构仅作为旧样例参考，不作为新规范延续

### 9.5 解析与判定原则

涉及 LLM/VLM 与自动判定的能力，统一遵守以下约束：

- LLM 不负责算术计算、阈值比较或确定性规则执行
- LLM 不负责最终合规判定，最终规则结论由工具侧逻辑输出
- 关键数字与关键字段的抽取结果必须保留置信度或来源追踪信息
- 所有问题结论都应能回溯到资料片段、规范条款或规则依据
- 第一阶段优先保证“证据可追溯”，再逐步提高“自动化深度”

## 10. 实施顺序

### 阶段 A：基线冻结与清场准备

- 盘点旧资产与保留资产
- 形成“旧到新映射说明”
- 明确目录删除清单
- 保留数据样例和脚本资产

### 阶段 B：新骨架建立

- 初始化 monorepo
- 建立 `apps/`、`packages/`、`tests/` 新结构
- 建立统一环境配置、开发脚本和文档入口

### 阶段 C：领域层重建

- 重写 P1/P2 核心 schema
- 迁移同义词映射和项目注册信息
- 明确新任务模型与结果模型

### 阶段 D：P1 接入

- 收编旧扫描脚本
- 建立 API → worker → 结果落盘闭环
- 完成 P1 结果展示页

### 阶段 E：P2 接入

- 以中文标准目录重写整理流程
- 建立 API → worker → 映射记录闭环
- 完成 P2 结果展示页

### 阶段 F：统一工作台接入

- 完成首页、模块导航、任务面板
- 把 P1/P2 真任务接入前端
- 为 P3-P5 建立骨架页

### 阶段 G：旧主线退出

- 删除旧目录
- 更新 README 与架构文档
- 明确后续 P3-P5 迭代入口

### 阶段完成标志

- 平台骨架阶段完成标志：能进入统一首页、模块导航、案例入口，并能通过前后端链路触发真实任务
- P1 完成标志：能完成一次规范扫描任务，查看状态、摘要与产物
- P2 完成标志：能完成一次项目整理任务，查看路径映射、警告信息与结果摘要
- 首阶段总完成标志：统一工作台可展示，P1/P2 可运行，P3-P5 有明确骨架与扩展位

## 11. 测试与验收标准

必须覆盖以下场景：

1. 前端、API、worker 可独立启动
2. P1 可创建任务、执行任务、查询状态、查看产物
3. P2 可创建任务、执行任务、查询状态、查看路径映射
4. 同义词映射能正确识别关键目录变体
5. 至少 1 个现有样例项目能跑通新 P2 链路
6. 前端能展示任务状态、结果摘要和产物入口
7. P3-P5 骨架页存在且不误导用户
8. 新工程不再依赖旧 `src/setri/cli.py` 或旧 `web/`

测试类型包括：

- 领域模型单测
- P1/P2 管道单测
- API 集成测试
- 前端基础交互测试
- 样例数据回归测试

## 12. 风险与应对

### 风险 1：误把旧中间产物当成可持续骨架

应对：

- 明确“迁移知识资产，不迁移旧骨架”
- 所有旧代码先做保留/废弃分类，再迁移

### 风险 2：P1/P2 与演示工作台双线推进导致范围失控

应对：

- 第一阶段只做 P1/P2 真能力
- P3-P5 只保留骨架，不提前实现

### 风险 3：旧数据结构与新目录规范冲突

应对：

- 中文标准目录作为唯一目标规范
- 历史英文结构仅作为样例输入和回归参考

### 风险 4：首阶段引入数据库与复杂基础设施拖慢进度

应对：

- 默认文件系统 + JSON 元数据起步
- 为数据库预留接口，但不作为阻塞项

---

## 13. 结论

本次重构的本质不是“继续开发旧原型”，而是：

**保留少量有效知识资产，清场旧中间产物，建立一套新的统一交付演示系统工程基线。**

第一阶段的完成标志是：

- 新 monorepo 工程骨架建立
- 统一工作台可展示
- P1/P2 真实可运行
- P3-P5 有清晰扩展位
- 旧 CLI/占位前后端骨架退出主线

在此基础上，后续迭代再逐步引入：

- P3 专家知识库
- P4 交叉提炼
- P5 自动审查
- AI 助手、RAG、知识图谱、专题扩展能力

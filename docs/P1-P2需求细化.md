## 1. 文档定位

本文是系统级 `PRD` 和重构计划的补充细化文档，只回答以下问题：

- `P1` 要做什么，为什么要拆成两种模式
- `P1/P2` 的输入对象、输出对象和结果层次应该如何定义
- `P1/P2` 的产物如何在后续被 `P3/P4/P5` 消费
- 证据、版本、引用和追溯关系应如何提前固化

本文不展开以下内容：

- 前端页面交互设计
- API 路径、数据库表、队列和部署实现
- 具体 prompt、模型调用参数和工程目录实现细节
- `P3/P4/P5` 的完整实现方案

---

## 2. 总体范围与原则

### 2.1 P1/P2 在系统中的角色

| 能力链路 | 核心角色 | 对下游的价值 |
|------|------|------|
| `P1` 规范库构建 | 把规范源转成可版本化、可引用、可追溯的条款资产 | 为 `P4` 规则形成、`P5` 规则依据、`P3` 规范参照提供底座 |
| `P2` 文件整理与解析 | 把项目资料转成标准目录、映射记录和统一解析记录 | 为 `P3` 知识抽取、`P5` 规则执行和证据定位提供底座 |

### 2.2 总体原则

- `P1/P2` 不只服务首期演示页面，而要为后续 `P3-P5` 预埋稳定契约。
- 证据优先于结论。所有自动结果都必须能回溯到文件、页码、片段或等价证据锚点。
- 版本先于优化。即使后续模型和规则会变，条款、解析记录和任务结果也必须先具备版本追溯能力。
- 受控输入优先于自由输入。首期宁可缩小模式和枚举范围，也不允许把高价值字段做成无法治理的自由文本。
- 结果对象优先稳定外层协议，内部 `fields` 允许按资料类型逐步丰富。

### 2.3 首期面向下游的最低要求

首期虽然不实现 `P3/P4/P5` 真能力，但 `P1/P2` 输出必须满足以下最低复用要求：

- `P1` 输出可被引用的条款资产和场景条款子集
- `P2` 输出可被引用的文件记录、字段记录和证据锚点
- 所有结果都能说明“来自哪次任务、哪一版上游资产”

## 3. P1 规范库需求细化

### 3.1 P1 的目标

`P1` 的目标不是只做“规范扫描”和“摘要展示”，而是形成两层能力：

1. 规范总资产层  
   负责沉淀完整的规范条款资产，为未来规则体系提供统一来源。

2. 场景消费层  
   负责面向特定专题或业务场景，输出高相关条款子集和规则种子提示。

因此，`P1` 固定拆成两种任务模式。

### 3.2 P1 双模式定义

#### 模式一：`full_coverage`

定义：

- 对规范源做全覆盖解析
- 输出完整条款库和基础冲突候选
- 形成某一专题下可版本化的规范总资产

用途：

- 作为 `P4` 规则形成的主来源
- 作为 `P5` 审查结论回溯时的规范依据池
- 为 `P3` 在解释专家意见时提供规范参照

#### 模式二：`targeted`

定义：

- 按专题或业务场景做定向解析
- 输出该场景下可消费的条款子集、主题摘要和规则种子提示

用途：

- 为 `P4` 形成场景规则提供快捷入口
- 为 `P5` 提供场景化规则配置基础
- 为前端工作区展示“该场景下最相关的规范依据”提供直接结果

约束：

- `targeted` 不是另一套孤立规范库
- `targeted` 必须引用同一套 `clause_id` 和 `regulation_version`
- `targeted` 本质上是从全覆盖规范资产中切出的场景化消费视图

### 3.3 P1 输入对象

P1 任务输入对象至少包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `mode` | 是 | `full_coverage` 或 `targeted` |
| `subject` | 是 | 规范主题名称，例如“配电网开关站” |
| `slug` | 是 | 规范库标识，用于资产归档与版本追踪 |
| `source_dir` | 否 | 规范源目录；默认指向仓库内标准规范目录 |
| `keywords` | 是 | 扫描和优先召回用关键词集合 |
| `profile_id` | 条件必填 | 仅 `targeted` 模式必填，表示专题/场景画像 |
| `profile_scope` | 条件必填 | 仅 `targeted` 模式必填，描述本次场景边界 |

输入约束：

- `profile_id` 首版采用受控枚举，不接受自由文本
- `profile_scope` 用于解释场景范围，不替代受控枚举
- `keywords` 用于扫描和初筛，不应被定义为唯一召回条件

### 3.4 P1 首版场景画像

首版建议至少预留以下受控场景画像：

| `profile_id` | 对应场景 |
|------|------|
| `switch_station` | 开关站专项审查 |
| `residential_redline` | 住宅红线内接入审查 |
| `tech_economic_review` | 技经资料预审与评审 |

说明：

- 本表用于约束 `targeted` 模式输入
- 后续可扩展新画像，但不应在实现时临时引入未治理的新值

### 3.5 P1 输出层次

P1 输出固定分为三层。

#### A. 任务结果层

用于任务中心、结果页和产物入口展示。

至少包含：

- `result_id`
- `task_id`
- `mode`
- `subject`
- `slug`
- `profile_id`
- `summary`
- `artifact_refs`
- `generated_at`
- `regulation_version`

#### B. 条款资产层

用于沉淀可被规则、知识库和审查结果引用的规范条款。

每条条款至少包含：

| 字段 | 说明 |
|------|------|
| `clause_id` | 全局稳定条款标识 |
| `regulation_version` | 该条款所属规范资产版本 |
| `source_id` | 来源规范标识 |
| `source_file` | 来源文件 |
| `source_page` | 来源页码 |
| `category_id` | 受控分类标识 |
| `topic_tags` | 主题标签集合 |
| `profile_tags` | 场景标签集合 |
| `subject` | 条款主题 |
| `text` | 条款正文 |
| `strength` | 强度词或约束强度 |
| `evidence_ref` | 指向规范证据的统一引用 |

补充要求：

- `category_id` 采用受控分类表，不允许自由生成
- `topic_tags` 负责跨专题检索
- `profile_tags` 负责跨场景召回
- `evidence_ref` 至少能回溯到 `file + page + snippet`

#### C. 场景消费层

用于场景化规则形成和审查消费。

至少包含：

- `profile_id`
- `selected_clause_refs`
- `excluded_clause_refs`
- `profile_summary`
- `rule_seed_hints`

说明：

- `selected_clause_refs` 表示该场景下优先消费的条款引用集合
- `excluded_clause_refs` 用于显式记录被排除但曾被召回的条款集合
- `rule_seed_hints` 不是完整规则，仅是后续 `P4` 可消费的规则种子提示

### 3.6 P1 证据与版本要求

P1 必须支持以下追溯能力：

- 能说明一条条款来自哪份规范、哪一页、哪次生成版本
- 能说明某个场景子集引用了哪一版规范总资产
- 能说明某条规则种子提示来源于哪些原始条款

因此，P1 至少要保留：

- `regulation_version`
- `source_file`
- `source_page`
- `evidence_ref`
- `artifact_refs`

### 3.7 P1 对 P3-P5 的复用要求

#### 面向 P3

- P1 不是专家知识库的主数据源
- 但 P1 必须为专家意见提供规范依据候选集
- 专家意见后续引用规范时，应能稳定引用 `clause_id`

#### 面向 P4

- `full_coverage` 是规则形成的总资产底座
- `targeted` 是场景规则形成的快捷入口
- `rule_seed_hints` 应为规则草案生成提供最小提示输入

#### 面向 P5

- P5 的审查结论必须可回溯到规范条款
- 因此 P1 必须提供稳定版本、稳定条款标识和证据引用
- P5 场景化配置应优先复用 `profile_id` 和 `selected_clause_refs`

## 4. P2 文件整理与解析需求细化

### 4.1 P2 的目标

`P2` 不只负责把文件搬到标准目录，而要完成两层结果沉淀：

1. 目录与路径层  
   解决“文件从哪里来、被整理到哪里去”。

2. 解析与证据层  
   解决“文件里被识别出了什么、证据在哪里、下游如何引用”。

因此，P2 的定义应从“文件整理”升级为“文件整理与统一解析记录沉淀”。

### 4.2 P2 输入对象

P2 首版输入对象至少包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| `project_id` | 是 | 预注册样例项目标识 |

首版输入约束：

- 仅支持预注册样例项目
- 不开放任意本地目录导入
- 项目必须同时登记送审版与审定版来源

### 4.3 P2 输出层次

P2 顶层结果对象至少包含：

- `result_id`
- `task_id`
- `project_id`
- `project_version`
- `path_mapping`
- `warnings`
- `summary`
- `artifact_refs`
- `parse_record_refs`

其中，`P2` 的核心结果必须分为两部分。

#### A. 路径映射结果 `path_mapping`

用于记录源路径与目标路径关系。

每条映射至少包含：

- `source`
- `target`
- `file_count`
- `note`

用途：

- 支持整理结果回放
- 支持异常定位和人工复核
- 支持后续比对送审版/审定版的源文件关系

#### B. 统一解析记录 `parse_records`

用于承载后续 `P3/P5` 可直接消费的解析底座。

每条记录至少包含：

| 字段 | 说明 |
|------|------|
| `record_id` | 解析记录标识 |
| `project_id` | 项目标识 |
| `file_ref` | 目标目录中的文件引用 |
| `document_type` | 文件类型，例如图纸、估算书、可研报告、材料清册 |
| `discipline` | 专业或分项类型 |
| `version_stage` | `submitted` 或 `approved` |
| `parse_type` | 解析类型，例如文本抽取、表格抽取、图纸理解 |
| `page` | 页码 |
| `snippet` | 关键片段 |
| `anchor` | 证据锚点，复杂场景可扩展区域信息 |
| `confidence` | 解析置信度 |
| `fields` | 结构化字段字典 |
| `evidence_ref` | 指向统一证据对象 |
| `trace_ref` | 指向执行 trace 或任务日志 |

说明：

- `parse_records` 是统一外层协议
- 不同资料类型可在 `fields` 中逐步细化，不应在顶层拆成多套互不兼容接口

### 4.4 P2 中文标准目录

P2 首版应明确中文标准目录是唯一目标规范。

目录主结构固定为：

```text
设计文件/
  送审版/
    项目级/
    开关站/
    街坊站-1/
    街坊站-2/
    站内电缆/
    高压电缆/
    低压电缆/
    通信/
    信息采集/
  审定版/
    项目级/
    开关站/
    街坊站-1/
    街坊站-2/
    站内电缆/
    高压电缆/
    低压电缆/
    通信/
    信息采集/
```

项目级标准文件集合固定为：

- `设计委托书`
- `可研报告`
- `中标通知书`
- `供电方案审核单`
- `财务审核表`
- `内审意见`
- `投资汇总表`
- `主要设备材料清册`
- `其他依据`

分项标准子目录固定为：

- `工程图纸`
- `建筑图纸`
- `工程估算书`
- `材料清册`

### 4.5 P2 稳定资料集

首版 P2 的解析沉淀优先覆盖以下稳定资料类型：

- `可研/初设报告`
- `工程估算书/概算书`
- `图纸/材料清册`

首版要求不是把所有资料解析到最深，而是先把统一结果协议、证据锚点和记录引用稳定下来。

### 4.6 P2 证据与版本要求

P2 必须支持以下追溯能力：

- 能说明一个字段或片段来自哪份文件、哪一页、哪次任务
- 能区分送审版和审定版同类资料
- 能为后续 `resolution` 跟踪提供记录级引用基础

因此，P2 至少要保留：

- `project_version`
- `version_stage`
- `file_ref`
- `page`
- `snippet`
- `anchor`
- `evidence_ref`
- `trace_ref`

### 4.7 P2 对 P3-P5 的复用要求

#### 面向 P3

- `parse_records + evidence_ref` 是专家知识抽取和三元关联的共同底座
- 专家意见在定位到设计文件时，应优先复用 `file_ref` 和 `evidence_ref`

#### 面向 P4

- P2 提供规则消费所需的文件类型、资料位置、结构化字段和证据定位基础
- 规则在形成时可以引用 P2 的资料类型和字段模式，而不是直接耦合原始文件路径

#### 面向 P5

- P5 执行规则时必须能命中文件、页码、片段和字段
- 因此 P2 必须提供统一 `parse_records`，而不是只提供目录整理结果
- `submitted/approved` 区分必须在记录层可追踪

## 5. 面向 P3-P5 的统一复用契约

### 5.1 统一引用对象

为避免后续 `P3/P4/P5` 各自定义一套引用方式，`P1/P2` 应统一预留以下引用对象：

| 引用对象 | 来源 | 用途 |
|------|------|------|
| `clause_ref` | `P1` 条款资产 | 供规则、知识关联和审查结论引用规范依据 |
| `file_ref` | `P2` 文件整理结果 | 供知识抽取、规则执行和结果定位引用目标文件 |
| `evidence_ref` | `P1/P2` 证据对象 | 供结果页、规则命中和知识记录统一追溯证据 |

### 5.2 统一证据对象

统一证据对象至少应支持以下字段：

- `evidence_id`
- `source_stage`
- `file_ref`
- `page`
- `snippet`
- `anchor`
- `confidence`
- `artifact_ref`

最低要求：

- 普通文本资料至少支持 `file + page + snippet`
- 图纸和复杂版面应预留区域坐标或版面锚点扩展位

### 5.3 统一版本对象

为支持回放、解释和规则升级，至少需要区分两类版本：

- `regulation_version`：P1 规范资产版本
- `project_version`：P2 项目资料资产版本

下游结果在未来至少应能记录：

- 消费了哪一版规范资产
- 消费了哪一版项目资料资产
- 结果对应哪一次任务和哪一批产物

### 5.4 下游消费关系

| 下游能力 | 对 P1 的消费 | 对 P2 的消费 |
|------|------|------|
| `P3` 专家知识库 | 引用规范条款作为依据候选 | 消费解析记录、文件引用和证据锚点做知识抽取 |
| `P4` 交叉提炼 | 消费全量条款和场景条款子集形成规则 | 消费资料类型、字段与证据模式形成可执行规则输入 |
| `P5` 自动审查 | 消费规范条款、版本和场景配置作为规则依据 | 消费文件记录、字段记录和证据定位作为命中对象 |

## 6. 结果对象与验收口径

### 6.1 P1 结果对象最小要求

P1 结果对象至少要能回答：

- 本次任务属于全覆盖还是场景化解析
- 解析了哪些规范文件
- 产出了哪一版规范资产
- 条款如何被引用
- 场景子集和规则种子如何被下游消费

### 6.2 P2 结果对象最小要求

P2 结果对象至少要能回答：

- 项目资料被整理到哪里
- 哪些文件存在警告或异常
- 形成了哪些可被下游引用的解析记录
- 每条记录的证据与置信度在哪里
- 送审版与审定版如何区分

### 6.3 本文档的验收问题

本文档完成后，应能直接回答以下问题：

1. `P1` 两种模式的职责边界是否清晰  
2. `P1` 的条款资产和场景子集是否是同一套版本体系  
3. `P2` 是否已从“文件整理”升级为“文件整理 + 统一解析记录沉淀”  
4. `P1/P2` 的输出是否已具备 `P3/P4/P5` 可直接消费的最小字段  
5. 证据、版本和引用关系是否已提前固化，避免后续返工

## 7. 默认假设与非目标

### 7.1 默认假设

- 后续需求推演、方案编写和实现讨论阶段，默认以 `Codex GPT-5.4 high` 作为当前主力辅助模型。
- 上述模型假设用于当前研发协作默认值，不构成系统运行时对单一模型的强绑定要求。
- `P1/P2` 的实现可以随模型替换演进，但不能改变本文定义的输入输出契约。

### 7.2 本轮非目标

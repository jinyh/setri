# 统一交付演示系统 Milestone 计划

| 字段 | 内容 |
|------|------|
| 文档版本 | v0.2 |

## 1. 计划目标

本计划用于指导统一交付演示系统从需求收敛到演示交付的推进过程。计划重点不是科研全过程管理，而是确保系统能够以统一平台的形式承接 `合同/` 目录中的 8 份项目材料，并完成可展示、可讲解、可验收的交付。

## 2. 规划原则

- 先统一，再分专题
- 先主线闭环，再做专题增强
- 先证据链与可解释，再追求更深自动化
- 以真实项目资料包演示为主，以预置样例补充

## 3. 里程碑总览

| Milestone | 目标 | 建议周期 | 主要产出 |
|-----------|------|----------|----------|
| M1 | 需求冻结与案例范围确认 | 第 1-2 周 | 需求文档、能力映射表、演示案例清单 |
| M2 | 统一平台骨架与信息架构确定 | 第 3-5 周 | 模块结构、页面流转、演示骨架 |
| M3 | 核心主线能力完成 | 第 6-10 周 | 业扩评审与技经评审主线演示 |
| M4 | 专题能力完成 | 第 11-15 周 | 开关站知识库、进度管控、竣工核查专题演示 |
| M5 | 联调、验收与演示材料收口 | 第 16-18 周 | 联调版本、验收口径、讲解材料 |

说明：周期按相对周数给出，便于后续结合实际资源映射到正式日历计划。

## 4. 里程碑明细

### M1 需求冻结与案例范围确认

目标：

- 将 8 份项目材料统一归并为一个产品需求基线
- 明确公共底座与专题模块边界
- 选定首批真实演示案例

重点工作：

- 完成统一需求文档
- 建立项目能力与系统模块映射表
- 确认首版必须演示的业务闭环
- 确认每个专题模块至少一个代表性案例

完成标准：

- 需求范围不再按单项材料编号组织，而按系统能力组织
- 各模块边界明确，避免重复建设
- 演示案例清单可支撑后续所有模块验证

依赖：

- `合同/` 目录材料完整可读
- 业务侧认可统一平台而非独立项目堆叠

### M2 统一平台骨架与信息架构确定

目标：

- 建立统一演示系统的信息架构和页面主骨架
- 固化公共流程与各模块入口关系

重点工作：

- 定义首页、案例入口、模块入口和结果页结构
- 确定统一主线：导入、解析、核查、证据、结论
- 明确公共能力底座的承载方式
- 明确专题模块与公共模块的跳转关系

完成标准：

- 各模块共享同一套主线，不出现 5 套割裂流程
- 首页可清楚区分核心主线与专题入口
- 各模块的输入、处理、输出位置明确

依赖：

- M1 中需求边界和案例范围已冻结

### M3 核心主线能力完成

目标：

- 先跑通最能代表统一平台价值的两条核心主线

范围：

- 业扩项目智能评审
- 技经资料预审与评审

重点工作：

- 打通资料导入、解析、规则匹配、问题输出
- 打通技经字段抽取、逻辑校验和优化建议输出
- 建立统一证据链展示和结果页输出
- 支撑至少一个真实案例完整演示

完成标准：

- 能完成一个端到端主线演示
- 核心结论均可回溯到资料和规则依据
- 输出结果可用于评审场景讲解

依赖：

- M2 平台骨架已具备
- 首批案例数据可用

### M4 专题能力完成

目标：

- 在统一平台上挂载剩余专题能力，体现项目群完整性

范围：

- 开关站专家知识库
- 工程智慧进度管控
- 竣工资料智能核查

重点工作：

- 完成专家知识展示、查询、问答和更新记录演示
- 完成进度预警、风险证据链和纠偏仿真演示
- 完成竣工图谱构建、三类核查和问题定位演示

完成标准：

- 三个专题能力均能在统一平台中独立演示
- 专题结果均复用公共证据与结果表达方式
- 讲解时能清楚说明其与核心主线的关系

依赖：

- M3 公共能力和结果框架已稳定

### M5 联调、验收与演示材料收口

目标：

- 形成可对内汇报、对外验收的统一演示版本

重点工作：

- 全模块联调与问题修正
- 固化演示脚本、讲解顺序和亮点话术
- 准备验收口径、案例说明和结果截图
- 输出版本说明、演示手册和答疑要点

完成标准：

- 至少 1 条核心主线和 3 个专题模块可稳定演示
- 演示过程中输入、处理、结论前后连贯
- 验收时可说明每项研究成果在系统中的承载位置

依赖：

- M3、M4 均完成并通过内部评审

## 5. 推荐交付顺序

建议采用如下优先级推进：

1. 统一需求与案例范围
2. 统一平台骨架
3. 业扩评审主线
4. 技经评审主线
5. 开关站知识库专题
6. 竣工核查专题
7. 进度管控专题
8. 联调与验收材料

排序依据：

- 业扩评审和技经评审最能体现项目群共性
- 开关站知识库与竣工核查和核心主线耦合度较高，适合在主线后接入
- 进度管控业务形态相对独立，适合作为专题增强模块

## 6. 每个里程碑的交付物清单

| Milestone | 交付物 |
|-----------|--------|
| M1 | 需求文档、能力映射表、案例清单 |
| M2 | 平台信息架构、页面清单、流程图、模块边界说明 |
| M3 | 核心主线演示版本、核心案例结果页、主线讲解脚本 |
| M4 | 专题演示版本、专题案例结果页、专题讲解脚本 |
| M5 | 联调版演示系统、演示手册、验收说明、答疑清单 |

## 7. 风险与应对

### 风险 1：模块过多导致系统像“拼盘”

应对：

- 始终以统一主线为骨架
- 所有模块共享同一套结果表达方式
- 首页只突出核心主线与专题扩展，不按单项材料并排铺开

### 风险 2：真实资料质量不稳定

应对：

- 每个模块至少准备 1 套稳定样例
- 对真实案例与演示样例分层管理
- 对多模态抽取结果保留人工复核与兜底展示

### 风险 3：专题能力深浅不一

应对：

- 首版以“可演示”而非“全自动最优”作为完成标准
- 共性能力优先做深，专题能力优先做通

### 风险 4：验收时无法说明研究成果承接关系

应对：

- 在演示材料中单独准备“项目成果映射表”
- 每个模块明确对应哪些项目成果

## 8. 当前默认假设

- 计划周期按 18 周估算，后续可结合实际资源调整
- 首版目标是统一演示系统，不是正式生产系统
- 演示以真实项目资料包驱动为主
- 需求优先级高于算法完整度，讲解完整性优先于局部深度最优

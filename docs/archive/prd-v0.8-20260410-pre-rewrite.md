## §1 愿景与范围

### 1.1 产品定位

**不是**：全自动审批机器。
**而是**：专家的"火眼金睛"副驾驶——处理数据比对与法规检索，专家负责决策与定性。

### 1.2 核心假设

1. 10kV 开关站项目的评审逻辑具有高度规律性，可通过"规范条款 × 专家经验"交叉提炼出可执行规则
2. 50 个历史项目足以覆盖开关站评审的主要场景，验证管道可行性
3. 规范库全量提取（不限专题）可为未来扩展到其他项目类型打下基础

### 1.3 系统边界

- **仅限开关站**：当前版本只处理 10kV 开关站项目（含配套街坊站、电缆、通信、信息采集分项）
- **不含**：架空线路、电缆隧道、箱变等其他配电网项目类型
- **数据规模**：50 个脱敏历史项目（数据已有/即将获取）
- **送审版与审定版**：P5 自动审查以送审版为输入基准；历史项目的审定版在 P2 阶段完整整理和解析，供 P3 阶段追踪整改落实（resolution）使用

### 1.4 终极目标

专家上传一个项目文件包 → 系统自动输出结构化审查意见（每条意见溯源到规范条款或专家历史模式）→ 专家审核确认后导出评审意见书。

## §2 管道总览

系统以"数据管道"为主线，将从原始资料到审查意见的完整链路拆解为五个管道站（Pipeline Station）：

```
P1-规范库构建    P2-文件整理与解析    P3-专家知识库    P4-交叉提炼    P5-自动审查
┌──────────┐   ┌──────────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ ~15份规范  │   │ 50个项目文件包 │   │ 50个项目   │   │ "应然"     │   │ 新项目     │
│ 全量提取   │   │ 标准化整理     │   │ 三元关联   │   │  ×         │   │ 文件包     │
│ 结构化条款 │   │ VLM解析       │   │ 模式归纳   │   │ "实然"     │   │ → 审查意见  │
└─────┬────┘   └──────┬───────┘   └─────┬────┘   │ → 可执行规则│   └─────┬────┘
      │               │                 │         └─────┬────┘         │
      │               │                 │               │              │
      ▼               ▼                 ▼               ▼              ▼
  规范条款库       结构化项目数据      专家知识库        规则库         审查意见JSON
  (clauses)      (parsed_data)     (knowledge)      (rules)       (opinions)
```

### 管道依赖关系

| 管道站 | 输入 | 输出 | 依赖 |
|--------|------|------|------|
| P1 规范库构建 | ~15 份规范文件 | 结构化条款库 | **独立**，可最先启动 |
| P2 文件整理与解析 | 50 个项目文件包（送审版+审定版） | 标准化目录 + 解析结果 | **独立**，可与 P1 并行 |
| P3 专家知识库 | 50 个项目的正式评审专家意见 + P2 的解析结果 | 三元关联 + 模式归纳 | 依赖 P2（文件定位需要解析结果） |
| P4 交叉提炼 | P1 的条款库 + P3 的专家知识库 | 可执行规则库 | **P1 ∩ P3**，两者都完成后启动 |
| P5 自动审查 | 新项目文件包 + P4 的规则库 | 结构化审查意见 | 依赖 P4 |

```
时间线：
        ┌─── P1（规范库）───────────────┐
        │                              │
启动 ───┤                              ├─── P4（交叉提炼）─── P5（自动审查）
        │                              │
        ├─── P2（文件整理）──┐          │
        │                   │          │
        └───────────────────┴── P3 ────┘
                          （专家知识库）
```

### 3.2 提取策略

对每份规范文件执行全量条款提取，每条条款附加两个维度评估：

**重要性（importance）：**

| 等级 | 含义 | 示例 |
|------|------|------|
| critical | 强制性条款，违反即不合规 | "电缆沟封堵应采用防火封堵材料" |
| major | 重要推荐，不遵循需说明理由 | "变压器防震垫宜按2组/台配置" |
| advisory | 建议性条款，供参考 | "建议采用节能型变压器" |

**可检验性（verifiability）：**

| 等级 | 含义 | 示例 |
|------|------|------|
| auto | 可通过数值比较/关键词匹配自动检查 | "安全距离 ≥ 2000mm" |
| semi-auto | 需 VLM 辅助提取后自动判断 | "图纸中开关柜间距" |
| manual | 需人工判断，系统仅提示 | "设计方案合理性" |

### 3.3 规范更新机制

- **增量提取**：新版规范发布时，仅提取变更条款，与现有条款库合并
- **冲突检测**：新旧条款内容矛盾时自动标记，提请专家裁决
- **版本管理**：每条条款记录来源文件版本号，支持按时间点回溯有效条款集

## §4 P2-文件整理与解析

### 4.1 数据规模与分批策略

50 个开关站项目，分三批迭代：

| 批次 | 项目数 | 目的 |
|------|--------|------|
| 第 1 批 | 1-10 | 验证整理流程 + 解析管线，建立质量基线 |
| 第 2 批 | 11-30 | 扩大覆盖，发现边界情况，优化同义词表 |
| 第 3 批 | 31-50 | 全量完成，稳定管线 |

> 当前已有 3 个测试项目（崇明盛世新苑、奉贤悦景名筑、奉贤咉懿佳园），属于第 1 批。

> **迁移说明**：早期按 v0.6 英文目录结构整理的项目数据，需按 v0.7 中文目录结构重新整理，确保所有项目统一使用同一套目录规范。

### 4.2 文件整理

将原始项目文件包按标准化目录结构整理，**送审版和审定版均需完整整理**。核心逻辑：

1. **目录解析**：通过同义词表 + 模糊匹配，将原始目录名映射到标准文件类型
2. **文件复制**：复制（非移动）到标准化路径，保留源文件不动
3. **元数据记录**：`meta.json` 记录原始路径 → 标准路径的映射关系

**同义词映射表**（核心子集，完整版见 §8.4）：

| 标准概念 | 变体 |
|----------|------|
| 开关站 | KT站, 开关站电气一次 |
| 街坊站 | PT-x站, PML, 街坊站电气一次 |
| 高压电缆 | 内线电缆-10kV, 10kV电缆, （10kV电缆） |
| 低压电缆 | 内线电缆-0.4kV, 0.4kV电缆, （0.4kV电缆） |
| 通信 | 内线通信, （通信） |
| 信息采集 | 用电信息采集, （信息采集） |
| 可研报告 | 可行性研究报告, 初设报告, 设计报告, 说明书 |
| 工程估算书 | 工程概算书, 工程估算书(PDF版), 工程概算书（PDF版） |

**模糊匹配策略**：
- 全角/半角括号统一：`（）` ↔ `()`
- 分隔符容忍：空格、括号、连字符均可
- 阶段术语等价：可研 ↔ 初设、估算 ↔ 概算

> **PML 柜归类**：PML 柜本质上是街坊站的一种形态，在标准化目录中归入 `街坊站-N/`，不单独设立分项目录。

### 4.3 文件解析

对整理后的文件进行结构化解析，**送审版和审定版均需完整解析**，输出存入 `解析结果/` 目录。审定版的解析结果供 P3 阶段 resolution 追踪使用。

**解析类型：**

| 文件类型 | 解析方法 | 输出 |
|----------|----------|------|
| 工程图纸（PDF） | 多模态 VLM | 设备拓扑、标注尺寸、布局信息 |
| 材料清册（Excel） | 表格识别 | 设备名称、规格型号、数量、单价 |
| 工程估算书（PDF/Excel） | 表格识别 + VLM | BOM 清单、造价明细 |
| 可研报告（PDF） | VLM 文档理解 | 章节结构、关键参数（变压器容量/数量、电缆型号等） |

**VLM 技术路线**：
- **阶段 A**：Gemini 3.0 Flash API — 快速验证，积累 prompt 模板与评测集
- **阶段 B**：Qwen VL 微调 — 基于积累数据微调，内网 GPU 私有化部署，脱离外部 API

**三条红线**（LLM/VLM 使用原则）：

| # | 原则 | 说明 |
|---|------|------|
| 1 | LLM 不做算术 | 数值比较、阈值判断、造价计算通过外部工具执行 |
| 2 | LLM 不做最终合规判定 | 合规/违规判定硬编码在工具内部，LLM 仅调度和转述 |
| 3 | 关键数值设置置信度阈值 | VLM 提取的关键数值附带置信度，阈值定义见 §8.5 |

## §5 P3-专家知识库

### 5.1 核心数据结构：三元关联

对 50 个项目的历史评审意见逐条建立"专家意见 ↔ 设计文件位置 ↔ 引用规范条款"三元关联。

> **数据来源**：仅使用"正式评审专家意见"（`过程管理/正式评审专家意见/`）作为三元关联的输入。预审意见、收口意见、发文意见等不纳入 P3 处理范围。

```
┌──────────────┐       定位到        ┌──────────────┐       依据于        ┌──────────────┐
│   专家意见     │ ──────────────── → │   设计文件     │ ← ──────────────── │   引用规范     │
│              │                     │              │                     │              │
│ · 意见原文    │                     │ · 文件路径    │                     │ · clause_id  │
│ · 意见类型    │                     │ · 页码/章节   │                     │ · 条款原文    │
│ · 严重程度    │                     │ · 定位区域    │                     │ · 标准版本    │
│ · 整改建议    │                     │ · 原文片段    │                     │              │
│ · resolution │                     │              │                     │              │
└──────────────┘                     └──────────────┘                     └──────────────┘
```

### 5.3 构建方法

| 阶段 | 项目数 | 方法 | 产出 |
|------|--------|------|------|
| 种子标注 | 10 个 | 资深专家人工逐条标注 | ~200-300 条 ground truth |
| VLM 扩展 | 40 个 | Gemini 3.0 Flash 批量处理 + RAG 规范匹配 | ~1500-2000 条候选关联 |
| 人工校验 | 50 个 | 按置信度分级审核（≥0.85 快速确认 / 0.6-0.85 重点审核 / <0.6 重新标注） | 完整三元关联库 |

**VLM 扩展的两个并行子任务**：

- **子任务 A — 意见→文件定位**：利用 Gemini 3.0 Flash 百万级上下文，将单个项目全部文档页面（~50页图像）+ 全部评审意见一次送入，批量输出每条意见的文件定位
- **子任务 B — 意见→规范匹配**：RAG 检索（意见文本 → 向量索引 → Top-5 候选条款）+ LLM 验证最佳匹配

### 5.4 跨项目模式归纳

在 50 个项目的三元关联完成后，进行跨项目分析：

- **高频审查点识别**：统计出现频率 ≥ 30%（≥15/50 个项目）的审查意见类型
- **典型问题模式**：归纳"问题描述 → 对应文件类型 → 依据规范"的固定模式
- **专家差异分析**：不同专家对同类问题的意见措辞和严重程度差异

这些模式直接输入 P4 交叉提炼，作为"实然"侧的数据基础。

## §6 P4-交叉提炼

### 6.1 核心思路

将 P1 产出的规范条款库（"应然"——规范要求怎么做）与 P3 产出的专家知识库（"实然"——专家实际怎么审）交叉比对，提炼出可执行的审查规则。

```
"应然"（P1 规范条款库）          "实然"（P3 专家知识库）
┌─────────────────────┐       ┌─────────────────────┐
│ clause_id: C-0042    │       │ 高频审查点:           │
│ text: 电缆沟封堵应采  │       │  15/50 个项目出现     │
│ 用防火封堵材料       │       │  "堵漏剂→防水封堵"    │
│ importance: critical │       │  意见                 │
│ verifiability: auto  │       │                      │
└──────────┬──────────┘       └──────────┬──────────┘
           │                              │
           └──────────┬───────────────────┘
                      ▼
              交叉提炼结果：
              ┌─────────────────────────┐
              │ 强规则 — 规范有+专家验证   │ → 自动检查
              │ 弱规则 — 仅规范有/仅专家有 │ → 半自动提示
              │ 不出规则 — 无法自动化      │ → 跳过
              └─────────────────────────┘
```

### 6.2 三种提炼结果

| 类型 | 条件 | 处理方式 |
|------|------|----------|
| 强规则 | 规范有明确条款 **且** 专家历史中高频出现（≥30%） | 自动检查，直接生成审查意见 |
| 弱规则 | 仅规范有条款（专家未关注）**或** 仅专家有模式（无明确规范依据） | 半自动提示，标记为"建议关注" |
| 不出规则 | verifiability=manual **且** 专家意见高度个性化 | 不生成规则，仅作为知识库参考 |

### 6.3 可执行规则 Schema

```json
{
  "rule_id": "R-0012",
  "name": "落后工艺材料拦截-堵漏剂",
  "source": {
    "clauses": ["C-0042"],
    "expert_patterns": ["EP-0087", "EP-0088"],
    "strength": "strong"
  },
  "check": {
    "target_file": "可研报告",
    "method": "text_match",
    "parameters": {
      "keywords": ["堵漏剂", "预制式终端"],
      "scope": "full_text",
      "expected": "absent"
    }
  },
  "judgment": {
    "severity": "warning",
    "opinion_template": "依据{clause_ref}，建议将"{matched_text}"改为防水封堵材料"
  }
}
```

### 6.4 四种检查方法

| method | 说明 | 适用场景 |
|--------|------|----------|
| `auto_value` | 数值提取 + 阈值比较 | 安全距离、设备数量、电缆长度 |
| `vlm_measurement` | VLM 从图纸/文档中提取数值后比较 | 图纸标注尺寸、设备间距 |
| `table_lookup` | 从表格中查找特定字段并校验 | BOM 清单核查、造价依据版本 |
| `text_match` | 关键词/正则匹配 | 落后工艺拦截、签章检测 |

## §7 P5-自动审查

### 7.1 审查流程

```
新项目文件包上传
       │
       ▼
┌──────────────┐
│ ① 文件识别    │  同义词表 + 模糊匹配 → 标准文件类型
└──────┬───────┘
       ▼
┌──────────────┐
│ ② 文件解析    │  VLM / 表格识别 → 结构化数据
└──────┬───────┘
       ▼
┌──────────────┐
│ ③ 规则逐条检查 │  遍历规则库，对每条规则执行 check
└──────┬───────┘
       ▼
┌──────────────┐
│ ④ 意见生成    │  触发的规则 → 填充 opinion_template → 结构化意见
└──────┬───────┘
       ▼
  审查意见 JSON
```

        "confidence": 0.95
      },
      "source_trace": {
        "clause_id": "C-0042",
        "expert_patterns": ["EP-0087"]
      }
    }
  ],
  "summary": {
    "total_rules_checked": 45,
    "triggered": 8,
    "by_severity": {"critical": 1, "warning": 5, "advisory": 2}
  },
  "confidence": 0.88
}
```

### 7.3 意见溯源

每条自动生成的审查意见必须可溯源到：
- **规则来源**：`rule_id` → 对应的规范条款（`clause_id`）和/或专家历史模式（`expert_pattern_id`）
- **证据定位**：触发规则的具体文件、页码、原文片段
- **置信度**：VLM 提取数据的置信度，按 §8.5 阈值分级处理

### 7.4 用户交互

专家在工作台上对每条意见执行操作：

| 操作 | 说明 | 数据记录 |
|------|------|----------|
| 采纳 | 意见转化为标准评审措辞，加入导出列表 | `{action: "accept"}` |
| 忽略 | 标记为误报，记录忽略原因 | `{action: "ignore", reason: "..."}` |
| 修改 | 基于意见内容编辑措辞后加入导出列表 | `{action: "modify", revised_text: "..."}` |
| 补充 | 专家手动添加系统未检出的意见 | `{action: "supplement", text: "...", file: "...", clause: "..."}` |

专家行为数据持续积累，反馈回 P3 专家知识库，形成闭环。

## §10 统一交互工作台

### 10.1 定位与目标

将 P1-P5 管道的全部交互从 CLI Skill（Claude Code）迁移到 Web 工作台，实现全流程可视化操作。工作台定位为**全流程工作台**（SaaS 级产品），评审专家和管理员通过浏览器完成从文件上传到审查意见导出的全部操作。

**核心价值**：
- 降低使用门槛，评审专家无需接触命令行
- 统一交互语言，所有管道站共享一致的 UI 组件和操作逻辑
- 任务可视化，实时追踪管道进度

### 10.2 用户角色

| 角色 | 职责 | 权限范围 |
|------|------|----------|
| 评审专家 | 上传项目、审核映射/条款、裁决冲突、审查意见确认 | P2-P5 操作、仪表盘、项目列表 |
| 管理员 | 管理规范库、配置管道参数、用户管理 | P1 操作 + 系统设置 + 评审专家的全部权限 |

### 10.3 架构方案

**管道导航型（Pipeline-centric）**：以 P1→P5 管道流程为主导航结构，左侧边栏列出五个管道站和管理功能入口。

> **演进规划**：当积累 50 个项目的知识后，未来过渡到"项目中心型"架构（以项目为核心组织单元，进入项目后看到各管道进度）。当前管道导航型便于以流程驱动知识积累。

**布局结构**：

### 10.4 交互模式：看板总览 + 步骤引导

每个管道站采用**混合交互**：

- **看板层（总览）**：管道站首页以看板形式展示所有任务/项目的处理状态（待处理 → 处理中 → 待审核 → 已完成）
- **详情层（步骤引导）**：点击看板中的卡片进入步骤引导页，按固定步骤推进操作

**步骤引导复用框架**：所有管道站共享 `StepBar` + `StepContent` 组件，仅步骤数量和内容不同。

### 10.5 交互原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | AI 先行，人工兜底 | 高置信度（≥0.85）自动采信，中低置信度标亮等待人工确认（阈值引用 §8.5） |
| 2 | 进度可视 | 步骤条 + 统计卡片 + 进度百分比，用户随时知道"到哪了" |
| 3 | 可回溯 | 步骤可回退，修改历史有审计记录 |
| 4 | 批量操作 | 审核表格支持全选、批量确认、批量跳过 |
| 5 | AI 理由透明 | 每个 AI 决策旁展示判断理由和置信度分数 |

### 10.8 数据流

```

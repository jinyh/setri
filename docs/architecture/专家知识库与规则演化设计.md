## 1. 定位

专家知识库不是自动审查的唯一主引擎，而是规则体系的校准与反馈层。它承担四类职责：

- 沉淀专家意见、规范依据和设计文件定位之间的关联资产
- 形成高频问题模式和解释素材
- 为规则生成与修订提供经验信号
- 承接专家对规则命中结果的结构化反馈

默认前提：

- P5 主链路是“规则库直接评审设计文件”
- P3 专家知识库不替代规则库对设计文件的主判定
- 专家意见匹配结果主要用于校准、解释和回溯

## 2. 核心知识资产

### 2.1 三元关联

P3 的基础资产是三元关联：

- 专家意见
- 设计文件位置
- 规范条款或规则依据

三元关联用于回答“专家为什么提出这个问题”“它指向哪份资料”“背后依据是什么”。

### 2.2 模式资产

在三元关联之上继续沉淀：

- 高频审查点
- 典型问题模式
- 常用意见措辞
- 同类问题的严重程度差异
- 规则修订建议

### 2.3 反馈资产

来自 P5 与工作台的人机协作记录应结构化入库，包括：

- 命中后被采纳的规则
- 被忽略的误报
- 被修改的表达或严重级别
- 专家手动补充的漏检问题
- 对规则范围、触发条件和例外项的修订请求

## 3. 数据来源与约束

专家知识库的数据来源包括：

- 正式评审专家意见
- P2 提供的结构化解析结果和证据定位
- 规范条款与规则版本信息
- 专家在工作台中的采纳、忽略、修改、补充行为

默认不将预审、收口、发文等全部文本混合为同一知识层；不同来源应保留来源标签和适用阶段，避免把噪声经验直接当作通用规则。

## 4. 从知识到规则的演化链

规则演化不应只依赖“专家怎么说”，而应同时消化三类输入：

- 规范条款：定义应然约束
- 真实设计文件命中结果：定义系统当前能稳定执行的判定基础
- 专家知识资产：提供高价值经验信号、语言表达和边界修正

### 4.1 规则生成输入

一条候选规则至少应回答：

- 依据了哪条规范或业务约束
- 需要从什么资料中提取什么字段
- 需要哪个工具执行最终判断
- 命中后输出什么严重级别和结果模板
- 需要什么例外条件或人工确认点

### 4.2 规则演化闭环

规则演化采用固定闭环：

1. 初版规则基于规范条款和可执行检查方法生成（Markdown 源文件）。
2. 规则编译：将 Markdown 源文件编译为 JSON 执行格式。
3. 在真实设计文件上运行，记录命中结果与执行 trace。
4. 专家在工作台执行采纳、忽略、修改、补充。
5. 系统按误报、漏报、表达不当、范围不准分类反馈。
6. 规则维护者修订规则条件、模板、适用范围或例外项（更新 Markdown 源文件）。
7. 重新编译规则，新规则版本进入回归验证，通过后替换旧版本。

## 5. 最小对象定义

### 5.1 三元关联对象

最小字段建议包括：

- `opinion_id`
- `project_id`
- `opinion_text`
- `source_stage`
- `document_ref`
- `evidence_ref`
- `clause_ref`
- `rule_ref`
- `confidence`
- `review_status`

### 5.2 专家反馈对象

最小字段建议包括：

- `feedback_id`
- `result_id`
- `rule_id`
- `action`
- `reason`
- `revised_text`
- `severity_override`
- `scope_fix_request`
- `created_by`
- `created_at`

其中 `action` 统一使用：

- `accept`
- `ignore`
- `modify`
- `supplement`
- `rule_fix_request`

### 5.3 规则对象（Rule）

规则对象采用 Markdown 源文件 + JSON 编译格式的双层表达：

#### Markdown 源文件（source/）

规则源文件使用 Markdown 格式，包含 frontmatter 元数据和规则内容：

```markdown
---
rule_id: R-KGZ-EQ-001
rule_name: 变压器容量校验
version: 1.0.0
category: 开关站/设备选型
complexity: simple | complex
execution_mode: structured | llm_reasoning
status: active | draft | deprecated
created_at: 2026-04-15
updated_at: 2026-04-15
author: 专家组
tags: [变压器, 容量, 开关站]
---

## 规则描述
...

## 规范依据
...

## 检查条件
...

## 结果模板
...
```

**字段说明**：

- `complexity`：规则复杂度
  - `simple`：输入字段明确、判断逻辑可用公式表达
  - `complex`：需要多模态解析、跨文档推理、语义理解

- `execution_mode`：执行模式
  - `structured`：编译为结构化 JSON，由规则引擎执行
  - `llm_reasoning`：保留 Markdown 引用，由 LLM Agent 推理执行

#### JSON 编译格式（compiled/）

**简单规则**编译为结构化 JSON：

```json
{
  "rule_id": "R-KGZ-EQ-001",
  "rule_name": "变压器容量校验",
  "version": "1.0.0",
  "category": "开关站/设备选型",
  "complexity": "simple",
  "execution_mode": "structured",
  "check_logic": {
    "input_fields": [
      {
        "name": "transformer_capacity",
        "type": "number",
        "unit": "kVA"
      }
    ],
    "calculation": {
      "required_capacity": "(load_demand / power_factor) * reserve_factor"
    },
    "conditions": [
      {
        "expression": "transformer_capacity < required_capacity",
        "severity": "major"
      }
    ]
  },
  "result_templates": {
    "insufficient_capacity": {
      "severity": "major",
      "description": "变压器容量不足...",
      "suggestion": "建议调整..."
    }
  }
}
```

**复杂规则**保留 Markdown 引用：

```json
{
  "rule_id": "R-KGZ-CS-002",
  "rule_name": "工程量跨文档一致性校验",
  "version": "1.0.0",
  "category": "开关站/造价核查",
  "complexity": "complex",
  "execution_mode": "llm_reasoning",
  "rule_file": "source/开关站/03-造价核查/R-KGZ-CS-002.md",
  "required_capabilities": [
    "multimodal_document_parsing",
    "cross_document_entity_alignment",
    "numerical_extraction_and_comparison"
  ]
}
```

#### 规则结果对象（RuleResult）

规则执行后产生的结果对象：

```json
{
  "result_id": "RES-20260415-001",
  "rule_id": "R-KGZ-EQ-001",
  "rule_version": "1.0.0",
  "project_id": "C109HZ22A062",
  "check_time": "2026-04-15T10:30:00Z",
  "status": "non_compliant",
  "severity": "major",
  "description": "变压器容量不足...",
  "evidence": {
    "file": "开关站/设备清册/变压器清册.xlsx",
    "location": "第3行"
  },
  "suggestion": "建议调整至1000kVA或以上",
  "confidence": 0.95
}
```

详细的规则编写指南和模板参见 `data/knowledge/规则库/README.md` 和 `data/knowledge/规则库/templates/README.md`。

## 6. 版本与治理

专家知识库和规则库必须独立版本化，但保持可追溯关联：

- 规则需要知道自己引用了哪批规范、哪类知识模式和哪轮反馈
- 知识库需要知道哪些记录已经被吸收到某个规则版本
- 规则升级后，旧结果仍应可回放和解释

知识库治理重点不是“图谱看起来多复杂”，而是：

- 来源清楚
- 适用阶段清楚
- 反馈能闭环
- 规则能复盘

## 7. 成功标准

- 能从知识库追溯一条问题结论的资料依据与规范依据
- 能解释某条规则为何存在以及最近一次如何修订
- 能分辨哪些经验只是历史表达，哪些已经转化为规则
- 能基于误报漏报和专家反馈持续改进规则命中质量

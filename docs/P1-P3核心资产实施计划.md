## 摘要

本计划经评估后修订，核心调整：

- **P1**：从单一模式改为双模式（`full_coverage` + `targeted`），补充新版 ClauseAsset 字段（regulation_version、category_id、topic_tags、profile_tags、evidence_ref）
- **P2**：从"P2-lite 最小支撑层"升级为完整两层输出（path_mapping + parse_records）
- **P3**：从"真实能力实现"降级为"骨架 + 种子样本"，标注为后续迭代目标
- **优先级**：P1 完整闭环 → P2 基础闭环 → P3 骨架

---

## 一、计划评估结论

**原计划需大幅修订。**

核心差距：

1. 原计划基于旧版 PRD v0.8（管道式架构），现行 PRD 已重构为"统一交付演示系统"
2. P1/P2需求细化 v0.2 定义了更完整的字段体系和双模式支持，原计划未对齐
3. P3 在新版 PRD 中降级为"首期仅产品骨架"，原计划按真实能力设计
4. 现有代码实现状态与原计划假设不符（P1 clauses = 0，P2/P3 未实现）

---

## 二、关键差距清单

### Gap 1: P1 条款字段定义

| 项目 | 原计划定义 | 新版需求细化 | 影响 |
|------|------------|--------------|------|
| 字段结构 | `source: dict` 嵌套 | `source_file, source_page, source_id` 平铺 | 引用方式不兼容 |
| 强度属性 | `importance` + `verifiability` | `strength` (约束强度) | 术语体系变化 |
| 版本管理 | `version` (规范年份) | `regulation_version` (资产版本标识) | 版本追溯缺失 |
| 分类标签 | 无 | `category_id` + `topic_tags` + `profile_tags` | 检索能力缺失 |
| 证据引用 | 无 | `evidence_ref` (统一证据对象) | 证据追溯缺失 |
| 任务模式 | 单一模式 | `full_coverage` + `targeted` 双模式 | 场景化消费缺失 |

### Gap 2: P2 输出层次定位

| 项目 | 原计划定义 | 新版需求细化 | 影响 |
|------|------------|--------------|------|
| 整体定位 | "P2-lite 最小支撑层" | "文件整理 + 统一解析记录沉淀" | 定位偏低 |
| 输出结构 | 仅 `EvidenceRef` | `path_mapping` + `parse_records` 两层 | 输出不完整 |
| 解析记录 | 6 个字段 | 16+ 字段完整结构 | 字段缺失严重 |
| 版本区分 | `version_stage` | `version_stage` + `project_version` | 项目版本缺失 |

### Gap 3: P3 实施优先级

| 项目 | 原计划定义 | 新版 PRD v1.0 | 影响 |
|------|------------|---------------|------|
| 首期定位 | 真实三元关联 + patterns.json | "首期仅产品骨架，不实现真能力" | 过度实现 |
| 数据来源 | 3 个项目正式评审意见 | 未定义首期范围 | 范围不明 |

### Gap 4: 现有代码实现状态

| 模块 | 现有状态 | 原计划假设 | 差距 |
|------|----------|------------|------|
| P1 pipeline | 有骨架，但 clauses = 0 | 可直接导出条款库 | 条款抽取未实现 |
| P2 evidence | 无实现 | 可从 `data/projects` 抽取证据 | 需新建模块 |
| P3 pipeline | 无实现 | 可构建三元关联 | 需新建模块 |

**关键发现**：`data/regulations/kaiguanzhan/regulations.json` 中 `total_clauses: 0`，条款抽取尚未完成。

## 三、修订实施计划

### 修订要点概览

| 修订项 | 原计划 | 修订后 |
|--------|--------|--------|
| P1 条款字段 | 旧版 schema (6 字段) | 新版 schema (12+ 字段) |

### Phase 1：完成 P1 条款抽取（优先级最高）

**现状**：`data/regulations/kaiguanzhan/regulations.json` 中 `total_clauses: 0`

**目标**：对 15 份规范 PDF 进行全量条款提取，生成非空 clauses.json

**关键文件**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/setri/common/schemas.py` | 修改 | 补充 ClauseAsset 新版字段 |
| `src/setri/p1_regulations/extract.py` | 创建 | 条款抽取主逻辑（VLM/API 调用） |
| `src/setri/p1_regulations/pipeline.py` | 修改 | 增加 run_extract 函数 |
| `src/setri/cli.py` | 修改 | 增加 `p1 extract` 命令 |
| `src/setri/config.py` | 修改 | 增加 LLM 配置常量 |

**新版 ClauseAsset 字段**：

```python
class ClauseAsset(BaseModel):
    clause_id: str
    regulation_version: str  # 新增
    source_id: str
    source_file: str
    source_page: int
    category_id: str = ""  # 新增
    topic_tags: list[str] = []  # 新增
    profile_tags: list[str] = []  # 新增
    subject: str = ""  # 新增
    text: str
    strength: str  # 替代 importance
    evidence_ref: str = ""  # 新增
```

**验收标准**：

- 运行 `uv run python -m setri.cli p1 extract --slug kaiguanzhan`
- 检查 `data/regulations/kaiguanzhan/clauses.json` 条款数 > 0
- 至少覆盖 3 个核心规范文件（技术原则、住宅导则、非居民导则）

2. `parse_records`：统一解析记录（首期最小字段集）

**首期 parse_records 最小字段**：

```python
class ParseRecord(BaseModel):
    record_id: str

    file_ref: str
    document_type: str  # 受控枚举
    version_stage: str  # submitted/approved
    page: int | None
    snippet: str = ""
    evidence_ref: str = ""
```

**关键文件**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/setri/common/schemas.py` | 修改 | 补充 ParseRecord, PathMapping |

**验收标准**：

- 运行 `uv run python -m setri.cli p2 parse --project C109HZ22A062`

## 五、实施建议
